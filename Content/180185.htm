Archiving General Ledger Entries

FIS Quantum G/L archiving enables you to manage the potentially large size of the gl_entry database table and the consequent impact on system performance.

Restrictions on Data Archived

Only G/L entry data not required for further processing can be archived. In particular:

  • deals must have matured - that is, their Maturity Date must precede the specified Archive Date, and
  • the G/L must be CLOSED - that is, the specified Archive Date must be the same as or before the last Close Date for the ledger.

    TIPS:

    • When a G/L is closed, the G/L Balance records for the G/L Owner are updated. This summarized balance information is required for historic G/L reporting (and improves performance).
    • When a G/L archive run is completed, a G/L Balance record for the archived G/L entries is also added.

Possible after-effects

The following Quantum applications are affected if any archived G/L entries are included in processing for the run (or save).

TIP: In each case, a validation message will be added to the Message Log.

G/L Close

Runs cannot proceed.

Reapply Cash Flows and Accounting

Runs cannot proceed.

Dealing

No deal actions are allowed and no entries show on the G/L Entries tab.

Reporting and Processing

Any archived entries are excluded from query result listings for Accruals, Currency Translation, Futures Margin, General Ledger Transactions, or Position reporting and processing runs.

The financial data is not compromised because:

  • the G/L Balance records cover archived entries, and
  • only entries for matured deals and closed ledgers can be archived.

TIP: If you wanted query results to include archived entries, you would need to restore the archived entries (by specifying the RESTORE parameter).
