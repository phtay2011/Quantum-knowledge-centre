Charges function (Settlement Processing)

Use this Quick menu function to change how charges are to be allocated for a particular EFT payment using SWIFT or a SWIFT-like Payment Method. The default setting is that specified for the Counterparty Bank Account.

Select one of the following (where valid for the relevant Payment Method):

NONE

Excludes Field 71A from EFT export files created for this Counterparty (not valid in many cases - for example, for SWIFT-style Payment Methods when Payment Category is Standard Settlement Instructions

BEN

All transaction charges are to be borne by the Beneficiary customer (see Note below)

BEN;BEN

Same as BEN (converted to BEN for SWIFT output; see Note below)

BEN;OUR

For SWIFT output: same as SHA (converted to SHA)

In other cases, opposite to SHA: charges on the Sender's side are to be borne by the beneficiary customer; those on the Receiver's side by the ordering customer

CLM

Charges are for the account of the bank claiming reimbursement

OUR

Charges are to be borne by the ordering customer (see Note below)

OUR;BEN

Same as SHA (converted to SHA for SWIFT output)

OUR;OUR

Same as OUR (converted to OUR for SWIFT output)

SHA

Shared - transaction charges on the Sender's side are to be borne by the ordering customer; those on the Receiver's side by the beneficiary customer

NOTE: When Detail of Charges is changed, it is considered a Modified SSI, even if it is subsequently changed back to its original value.

For more information, see MT101/103/103+ Counterparty Mapping.

  • Where the EFT Processing page is used to output SWIFT Payment messages (and provided you have sufficient Module Security rights*), a Bank Account Adjustment will be automatically created (by the EFT Payments Interface program) when the payment is output, to reflect the charges applied (BEN or OUR only).
  • The required Module Security rights are Add and Bank Mode for Bank Account Adjustments. If you do not have these, a Security Access message is written to the Message Log.
