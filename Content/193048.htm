Using Expression Builder - Functions

In Quantum, functions in Expression Builder are predefined commands whose arguments are evaluated in a specific order. They provide the foundation for:

  • setting up calculated columns, for example in queries* within the Configure View panel
  • evaluating the validation conditions in workflow pages created on a "Variant", for example, when applying cut-off date validation.
  • setting up the Configure View panel's Columns tab and Colors tab for any reports
  • aggregating data based on the current context (the type of data displayed) and use that aggregated data in an expression. In particular, use the aggregation functions in an expression for a workflow report's Calculated Columns area.

Arguments may be:

  • field names enclosed in square brackets - for example [Instrument];
  • text or Boolean literal values enclosed in double quotes - for example "XYZ Bank", "FX", "true";
  • numeric values without delimiters - for example 5000000;
  • lists of items of the same type enclosed in curly brackets - for example {"AC", "FX", "MM"} and { [CashFlowDate], [PositionDate] }; or
  • nested functions returning a valid value for the argument.

*TIP: In Analytics queries, configure Calculated Columns at the query (not view) level.

Using curly brackets {}

You can select the aggregating functions from the Functions folder when configuring calculated columns, including:

  • In
  • Sum(value, optional groupBy)
  • Count(value, optional groupBy)
  • DistinctCount(value, optional groupBy)

These all use one of two general formats. Notice the effect of the curly brackets {} in the examples.

Sum( [Field to Aggregate], { [List of Fields to Group By], ... } )

OR

Sum({ [Field to Aggregate], [List of Fields to Group By], ... } )

You could use a Pivot to aggregate data, but using functions allows you to see the data in a table and see more layers of information.

Limits

The system uses Expression Builder in many areas, so there are some limits to how you can use it. For example:

  • When setting up Workflow Page or Workflow Report pages in the Enterprise Workflow (EW), you can use Expression Builder to select from fields that are in an existing details table. However, don't set up header tables to refer to sources that use detail tables: it won't work.

Abs

Description

Returns the absolute value of a number (the unsigned value)

Format

Abs(number)

Arguments

number- the number or column name whose absolute value you want

Example

Abs([Margin])

If Margin is -2.50, the result is 2.50.

AddBusinessDays

Description

Returns the date that is the given number of business days from the given date, where business days are based on the given currency and locations

Format

AddBusinessDays(date, days, currency, locations)

Arguments

date - the date to start from

days - the number of days to add (a negative number of days will go backwards from the starting date)

currency - the applicable currency

locations (optional) - one or more locations

Example

AddBusinessDays(Date(2019,5,31),3,"USD")

Results in 5 Jun 2019 since Saturday and Sunday are configured as a non-business day for the USD currency.

AnalysisCode

Description

Returns the value for the given Analysis Code Name, Scope, and key value

Format

AnalysisCode(scope, key, analysis_code_name)

Arguments

scope - the Scope defined for the Analysis Code: Deals, Instrument Type, or a supported transaction or static data type - see Additional Notes below

key- the key value, depending on the Scope: (Deal Number, Entity Name, and so on)

analysis_code_name - the Name identifying the specific Analysis Code

Example 1

AnalysisCode("FX","1234","Int")

Results in the value of the Int code for FX deal number 1234.

Example 2

AnalysisCode("Entity","OurCorp","Rating")

Results in the value of the Rating code for the OurCorp entity.

Additional Notes

The scope input parameter values may be:

  • in the same form as in the Scope drop-down on the Configuration > Analysis Code page,
  • a truncated form of transaction type (without the word Deal), or
  • a standard transaction type code like FX, CO, or MM.

The function is NOT currently supported for Analysis Codes with these Scope values:

  • Bank Account Transfers
  • Bank Accounts
  • Facility
  • Futures Deals
  • Hedge Relationship Deals
  • Letters of Credit Deals
  • Projected Cashflows
  • Repurchase Deals
  • Unit Deals
  • Unit Holdings

DealCutOffDate

Description

Returns the minimum deal cut-off date from today based on the best-matching Cut-Off Time rule with a Scope of Deal, where:

  • if cut-off time not yet passed - returns (Today + (Cut-off days) business days), where Today is based on the rule location and business days are based on the locations of the deal Entity, Counterparty, and Bank Account, and Currency;
  • if Cut-off time is passed, returns (Today + (Cut-off days +1) business days), where Today and business days as above; or
  • if no matching rule is found - returns Today's date at the user's location.

Format

DealCutOffDate(transactionType, dealType, instrument, counterparty, entity, currency, bankAccount, otherCurrency)

Arguments

NOTE: Each argument may also be a field name within square brackets (which will provide the actual value in that field at run-time) or a valid nested function.

transactionType - a valid Transaction Type to be matched (with valid abbreviations allowed): "FX", "Money Market", "UNH", and so on)

dealType - "Deal, "Exposure", "Quote", or "Request"

instrument - one Instrument name

counterparty - one Counterparty name

entity - one Entity name

currency - one Currency name

bankAccount (optional) - blank, "ALL" or one account name

otherCurrency (optional where applicable) - blank, "ALL" or one or more* Currency names

Example

DealCutOffDate("AT", "Deal", [Instrument], [Counterparty], [Entity], [Currency], [Bank Account.Account Number] )

DealCutOffDate("AT", "Deal", "Bank Account Transfers", "XYZ Corp", "SBU Entity", "GBP", "", )

If

Description

Returns one value if a specified condition evaluates to TRUE; another value if it evaluates to FALSE

Format

If(logical_test,value_if_true,value_if_false)

Arguments

logical_test - the conditional expression

value_if_true - the returned result if the condition is TRUE

value_if_false - the returned result if the condition is FALSE; must be the same type as the TRUE result

Example

If([Amount] > 10000000,[Amount],0)

If Amount is:

  • 9,900,000.00 - the result is 0,
  • 10,500,000.00 - the result is 10,500,000.00.

Additional Notes

  • The second and third arguments must result in the same type.
  • If statements can be nested - that is, the arguments in an If statement can be other If statements.
  • An If statement cannot contain a Join statement.

Join

There are two Join functions.

1. Join(values...)

Description

Returns a single text string of joined values.

Format

Join(values...)

Arguments

values - the text strings and/or numbers to be joined, in correct order, comma-separated

Example

Join([Currency],"/",[Entity],"-",[Instrument])

If Currency is AUD, Entity is OurCorp, and Instrument is AUD LT Borrow, the result is:

AUD/OurCorp-AUD LT Borrow


2. Join({values}, separator)

Description

Returns a single text string of joined values delimited by a specified separator.

Format

Join({values},separator)

Arguments

{values} - the text strings and/or numbers to be joined, in correct order, comma-separated, within curly brackets

separator - user-defined separator

Example

Join({[Comment], "Dare", "To", "Dream" }, " ")

If Comment is Always, the result is Always Dare To Dream with the separator being a space.

Join({[Currency],[Entity],[Instrument]},"-")

If Currency is AUD, Entity is OurCorp, and Instrument is AUD LT Borrow, the result is:

AUD-OurCorp-AUD LT Borrow

Additional Notes
  • The number of arguments is unlimited.
  • Arguments can be any other expression, field, or value.
  • A Join function is not allowed within an If function.
