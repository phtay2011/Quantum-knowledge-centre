Message Queues and Data Integration

In order to improve Quantum integration with external systems, you can now configure the import and export of data in Quantum using message queues.

Additionally:

  • Importation Services such as Deal Import, Static Data Import, and Confirmation Matching can now read from message queues;
  • outgoing information such as Scheduler, AGCommandLineSupport, and Paper Confirmations can now write to message queues;
  • outgoing information can be signed and incoming information can be verified using digital certificates; and..
  • outgoing information can be encrypted and incoming information can be decrypted using XML Encryption.

Implementation

A new Data Transfer Location page has been added to the Configuration menu. Use this page to configure data streams into and out of Quantum as queues and files at various integration points. Also supports data from the Quantum API.

This page includes the following key fields.

Details

Purpose - Select one of the following incoming or outgoing data types.

  • Incoming data*:
    • BankImportation - Bank Statement items (account balances and transaction flows),
    • Confirmation - Confirmation Matching messages returned by the external matching service,
    • DealImportation - deal details,
    • StaticImportation - Static Data, or
    • WorkflowIn (only when Location Type is FILE) - a input text file that will be parsed by the specified System Parser (if any are installed) into a Data Source.
  • Outgoing data:
    • ConfirmationDelivery - Confirmation messages output from the Confirmations Processing page,
    • Eft - EFT messages output from the EFTs Processing page OR EFTs Reprocessing Utility page (where the External Method Source for the specified Payment method has Type Export set to Create File),
    • GLInterface - General Ledger entries exported from the G/L Interface page,
    • Scheduler (only when Location Type is QUEUE) - required for Scheduler tasks wanting to use the Message Queue defined in this DTL record to output Quantum data , or
    • WorkflowOut - an output text file formatted as defined in the specified Workflow Template.

TIP: An Incoming or Outgoing badge displays at the top right to show the direction of your selected data type.

Location

Location Type - Select FILE to transfer data using the file-based interface; select QUEUE to transfer data using the MQ-based interface; or select WEB API to transfer data using the Quantum API.

Queue Connection - (Available only when Location Type is QUEUE or WEB API.)

Select the queue connection.

Queue connections are controlled via the MessageQueueSettings.config file, located in the \config directory of the Site folder. Queue connections are established using Apache.NMS.

Digital Signature

Digital certificates bind together the public and private keys required for asymmetric encryption and signing of digital content. Quantum utilizes certificates based on either X.509 standards (for XML encryption) or PGPKeyRing (for PGP). Contact your FIS Consultant for more information.

Configure these settings as required to apply digital signatures to transferred data.

Certificate Store - The location of the Certificate store in the Registry. The available settings depend on the selected Signature Type as follows:

  • for PGP - PGP Ring is the only available setting, or...
  • for CMS - select Local Machine or Current User.

Certificate Criteria - Enter the criteria, based on the selected Signature Type:

  • for CMS - enter the Certificate file name (without the file extension), or...
  • for PGP - enter the PGP Key ID.

Certificate Criteria Type - Select a Find By option for locating the Certificate file to apply. The available settings depend on the selected Signature Type as follows:

  • for PGP - Find By Subject Key Identifier, is the only valid setting; or...
  • for CMS - select one of the following:
    • Find By Application Policy,
    • Find By Certificate Policy,
    • Find By Serial Number,
    • Find By Subject Distinguished Name,
    • Find By Subject Key Identifier,
    • Find By Subject Name, or
    • Find By Thumbprint.

Detach digital signature - (Available only when a Signature Type is specified, and only for outgoing data - that is, when Purpose is Eft, ConfirmationDelivery, or GLInterface.)

Select this option to generate the digital signature as a separate file, OR clear the option to include the digital signature in the output file.

NOTES:

  • No Digital Signature will be applied if Certificate Store or Certificate Criteria Type is blank or none.
  • Certificates used for encryption must be available to each Quantum Application Server.
  • When signing or encryption is required for binary output the content is first base64 encoded and then converted to XML within an <encoded> element.

Configuration

Queue connections are controlled via MessageQueueSettings.config, located in the \config directory of the Site folder. Queue connections are established using Apache.NMS.

Quantum supports the following NMS implementations:

  • ActiveMQ
  • MSMQ
  • IBM WebSphere

NOTES:

  • The full list of MSMQ options can be found here: https://msdn.microsoft.com/en-us/library/ch1d814t(v=vs.110).aspx.
  • MSMQ has a size limitation of 4MB per message, which may not be suitable for all message outputs.
  • ActiveMQ is not installed with the core Quantum product, and clients are required to configure this themselves.
    Refer to the ActiveMQ setup documentation here: http://activemq.apache.org/version-5-getting-started.html#Version5GettingStarted-StartingActiveMQStartingActiveMQ.

Implementation Notes

Digital Signature and Encryption

Outgoing information can be encrypted and incoming information decrypted using XML Encryption. Both security features can be applied to file-based and MQ-based interfaces. XML Encryption is a standards-based, interoperable way to encrypt all or part of an XML document or any arbitrary data. The .NET Framework XML encryption classes implement the World Wide Web Consortium (W3C) specification for XML encryption - see http://www.w3.org/TR/xmlenc-core/.

The XML document encryption uses an X.509 certificate. Internally a symmetric session key is used to encrypt the document and then the X.509 certificate is used to embed an encrypted version of the session key into the XML document.

Incoming messages can be decrypted as part of importation process. Decryption will be done automatically based on the content of the message. Certificate information must be included in the incoming XML message.

Incoming Services

Data from importation services such as Deal Import, Static Data Import, and Confirmation Matching can be read from queues.

When using an incoming queue, the content will still be written to the \Processed directory, and any failed records will still be written to the error directory. In case of an MQ deal import failure, the context of the MQ import message can be saved in a file and import process could be completed via file-based interface.

TIP: Incoming files can also be decrypted and verified against embedded digital signatures.

NOTES:

  • While the initial content will be moved through to the processed directory, any items in the \Error directory will be in decrypted form.
  • Deal, Static Data and Bank Transaction Import queue messages require a header in the form of [<extension>], followed by the import program name (for example, [qxt]FXDeal).

Writing to the Queues

The key points to note are as follows:

  • The easiest way for external .NET applications to write to the queues is to use Apache.NMS.
  • All incoming services except queue content is text-based.
  • For MSMQ, text-based content should be in UTF32 format.
  • For ActiveMQ, since there are multiple formats which could also be in compressed form, NMS/JMS is recommended.

Outgoing Services

Outgoing information such as Scheduler, AGCommandLineSupport, EFTs, and Paper Confirmations can write to message queues. The key points to note are as follows.

  • Queues should only contain one type of content: either text-based or binary.
  • Scheduler, AGCommandLineSupport, and Paper Confirmations write Binary based content.

When binary content is to be signed or encrypted, it will be converted into a base64 string and saved as Text content.Reading from Queues

The key points to note are as follows:

  • The easiest way for external .NET applications to retrieve from the queues is to use Apache.NMS.
  • For direct access to MSMQ:
    • Content is accessible from the BodyStream.
    • Text-based content will be in UTF32 format.

      NOTE: Use System.Text.Encoding.UTF32.GetString to convert the content.

  • For ActiveMQ, since there are multiple formats which could also be in compressed form, NMS/JMS is recommended.

Message Header Handling

For certain MQ services there is a different approach for handling and location of the header of the message.

  • Microsoft Message Queuing or MSMQ - The information is stored in the System.Messaging. Message.Extension property (which is a byte[] and is a serialized hashtable) with a key Header.
  • Apache ActiveMQ - The information is serialized into the message body. However the only way to drive ActiveMQ is via the NMS provider so you can directly use message.Properties["header"] to specify the content.
