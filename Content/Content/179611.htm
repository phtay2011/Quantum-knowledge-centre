Importer fields for Linked Deals

A linked deal in a Hedge Relationship is identified by a unique combination of three keys:

  • Deal number
  • Transaction type
  • Start Date (the date from which the deal is linked to this Hedge Relationship)

Values are required in these fields when you add, modify, or delete a linked deal.

Default settings

As the deal to be linked is already in Quantum, other settings are generally defaulted from the underlying deal, or from Accounting Rule settings.

Fields that are mandatory on the Hedge Relationships form are mandatory when importing. If you attempt to import a linked deal that is incomplete, the record will be rejected.

Changing an existing setting

If a defaulted setting can be changed on the Hedge Relationships form, you can change it during importation.

For example, you can change the nature of risk for a Money Market deal; but you cannot change the instrument associated with a deal, and any record that attempts to change it will be ignored. See also Editing a linked deal using the Deal Importer.

Table of Importer fields for Linked Deals

The QID-format fields in the following table are supported for importing of linked deals into a Hedge Relationship. They correspond to columns on the Linked Deals tab, and are listed in the order in which they appear there.

TIP: For a description of supported fields for importing Hedge Relationships, see Importer fields for Hedge Relationships and Importing Hypothetical derivative deals

When using this table, note that:

  • If N/A appears in the Field code and Example columns, you cannot import a value for that field using the Importer - the value is either defaulted or calculated by Quantum.
  • Each line containing a group of these fields (a single linked deal) must be preceded by hrlinks= .
  • To enter multiple linked deals one after the other, simply enter a new hrlinks= line for each. Follow each of these lines with the appropriate command. See Linked Deal commands, Import files.
  • If a field is not applicable to the linked deal that you are importing, there is no need to include it in the hrlinks record.
  • See also Delimiter below.

NOTE: The same fields that are required on the Hedge Relationships form before a Hedge Relationship can be saved, are required here. If you attempt to import a linked deal that is incomplete, the record will be rejected.

For example, an Actual Cash flow requires an Interest Rate (reval_rate) setting whether it is entered using the Linked Deals tab or imported.

If this is the only Hedge (or Exposure), the default is the maximum percentage possible. This is the same as the Unallocated percentage that was shown for the underlying deal before the deal was selected.

component

Revaluation Method

component=FORWARD NPV

Each deal in a Hedge Relationship is revalued using the method defined for that deal type in Accounting Rules. Because there is the concept of risk components in Hedge Accounting however, the revaluation changes can be split between those risk components.

Use this field to specify which risk component is being hedged by the linked deal. This influences the values used for effectiveness calculations and postings.

Quantum generally provides a default based on the Transaction Type, Hedge Type, and settings in Accounting Rules. Actual and Projected Cashflows are the exception. They have no Accounting Rules defined, so you must provide a revaluation method here.

If Quantum is licensed to import market values, you can specify a revaluation method of IMPORTED for Actual and Projected Cashflows or Currency Options, and IMPORTED or IMPORTED SEP for FX Deals. See Hedge Accounting, imported market values.

See also Revaluation methods in Hedge Relationships

reval_ccy

Revaluation Currency

reval_ccy=BASE CCY

Specify the Revaluation Currency to be used when the deal is revalued as part of the Hedge Relationship. It is valid for the following deals.

  • Actual and Projected Cashflows

    There are no Accounting Rules defined for Actual and Projected Cashflows, so use this field to specify BASE CCY (the default) or NON BASE CCY. Quantum can only revalue an Actual or Projected Cashflow as part of a Hedge Relationship. The rules applied are similar to those used to revalue FX Deals (see below).

  • FX Deals

    The default is the Revaluation currency (FX Deals) setting specified in Accounting Rules. To change it, specify BASE CCY or NON BASE CCY as required.

    If Revaluations is set to NPV SEPARATE or IMPORTED SEP in Accounting Rules, this value is automatically set to SEPARATE. A different currency can then be used for each leg of the deal.

  • Currency Options

    Specify BASE CCY, NON BASE CCY, or PREMIUM CCY. The default is the Revaluation currency setting specified in Accounting Rules.

    It is always the premium that is revalued for a Currency Option. The specified revaluation currency is only used to post the unrealized gain or loss on revaluation. For a description of the options, see Revaluation Currency (Currency Options).

Specify another value to link only a portion of the deal, based on its term structure. For example, you could allocate only years two, three, and four of a ten-year swap to this Hedge Relationship.

The End Date must be the same as, or earlier than, the maturity date of the underlying deal. For a rolled deal, Quantum validates this using the revised End Date.

leg

Leg 1 / Leg 2

leg=1

Specify the leg used in the Face Value Hedge Ratio calculation and for automatic allocation. Both of these calculations require a common currency and an opposite sign between Exposure and Hedge. The Leg 1/Leg 2 setting also affects Position Inquiry calculations.

The first linked deal added should be an Exposure. This will automatically be set to Leg 1. When you add a Hedge, the currency associated with Leg 1 of the Exposure determines the default leg to be used for the Hedge.

NOTE: Changing this setting will cause other values to change, depending on the Transaction Type. For example, changing the leg for an Interest Rate Swap will cause the Leg 1 Pay/Rec setting to change.

clean_dirty

Clean / Dirty

clean_dirty=clean

Specify either Clean or Dirty pricing for Effectiveness Testing.

When Clean is selected, Quantum will:

  • Strip Income accruals from the revaluations for Effectiveness Testing within a Hedge Relationship (HR) deal (where supported - see below).
  • Show the Accrued Interest in Revaluation Analysis, for reconciliation purposes.

Clean Effectiveness Testing is supported for:

  • Money Market, Interest Rate Swap, and Currency Swap deals participating in the Hedge Relationship, where the Revaluation Method (Links tab) is Projected, Market Imported, or Imported Sep.
  • Fair Value and Cash flow Hedge types, where the Designated Risk is either Interest Rate Risk or Overall Market Risk (not Exchange Rate Risk).
  • Each or both legs of a Swap deal, for calculating Hedge Effectiveness and Dollar Ineffectiveness.

When Dirty pricing is selected, settled cashflows (coupons/interest amounts) are added back as the revaluation date moves past the cash flow date (on a Swap however, this occurs only if one leg is selected).

reval_rate

Interest Rate Name

reval_rate=Zero Blended Curve

The name of the interest rate used to revalue the underlying deal. Quantum uses this name and the rate type (in the next column) to identify the rate.

Generally defaults from Accounting Rules, but mandatory for Actual and Projected Cashflows.

NOTE: You can save a Hedge Relationship without a rate name only if the Component value is IMPORTED or IMPORTED SEP.

reval_type

Rate Type

reval_type=Daily Market Rates

The type of interest rate used to revalue the underlying deal. One rate can have multiple uses in Quantum, each use distinguished by a rate type.

Generally defaults from Accounting Rules, but can be changed. Mandatory for Actual and Projected Cashflows.

NOTE: You can save a Hedge Relationship without a rate type only if the Component value is IMPORTED or IMPORTED SEP.

Delimiter

The record for a linked deal contains:

  • hrlinks=
  • At least the three key items identified above (Deal Number, Transaction Type, and Start Date)
  • Any other parameters that are to be changed

To separate the fields within a linked deal, insert a tilde (~) delimiter between items, as shown in this example:

hrlinks=deal_no=2~transtype=FX~expo_hedge=H~reval_ccy=BASE CCY~component=FORWARD NPV~proportion=100~leg=1~link_st_dt=01-Dec-1998~link_en_dt=29-Oct-1999~reval_rate=Zero Blended Curve~reval_type=Daily Market Rates~margn_type=NONE~vol_rate=NONE~rskfree_rt=NONE~

This delimiter (~) ensures that fields are recognized as part of that single linked deal. Tabs or paragraph marks () are used to separate fields in the record.
