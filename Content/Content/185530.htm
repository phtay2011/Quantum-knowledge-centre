Page Customization

(First introduced in Release 6.3.1 SP, August 2016.)

You can now customize Dealing and Pricing pages as follows.

  • Pages
    • Customize a core page for all users.
    • Create one or more variants of a core page.
    • Control access to variants through module security.
    • Customize the URLs of variant pages.
  • Page elements
    • Restrict access to page elements such as fields, menu options, and tabs, by hiding or disabling them.
    • Customize the titles and field labels on core and variant pages.
  • Analysis Codes
    • Position analysis codes with other fields.
    • Dynamically hide analysis codes when other fields are hidden.

You can also:

  • control the visibility of deals in core and variant customizations of Dealing pages; and
  • export customizations - for example, for importing into other databases.

Customization Page

A new Customization page has been added to the Configuration menu. This page enables you to define the following using customization XML syntax:

  • Core page customization, in which pages that are part of the core product can be customized. Only one core page customization can be created for each core page and its URL cannot be changed.
  • Variant customization, in which multiple versions of core pages can be created and customized. The URL can be defined for variant pages.

NOTES:

  • To be able to see the Customization page, users will need to have Module Security permissions for the Customization module.
  • For both core and variant customizations the page title can be customized. This is reflected in the page title as well as the menu.

Customization XML Editor

The Workflow page's XML editor provides word completion for elements defined in the XML schema.

TIP: Press CTRL+SPACEBAR to display a list of possibilities within the editor.

The following can be defined in the XML editor:

  • Add lookup values and conditional defaults through scenarios. You can define fields using defaultValue and includedValues/excludedValues attributes. You can also define conditions. When the condition is met, the system applies the field settings under that condition. If the same field is matched under different scenarios, the first match is chosen per attribute. Since the includedValues and excludedValues attributes are both about the lookup values, these attributes are considered one definition. When a field only specifies one of them, it will use the default value for the attribute - "*" for include and "" for exclude.
  • Change field titles on a page. Changes to field titles are reflected in any basic validation messages that refer to the field and the title of the lookup dialog box (for reference fields).
  • Hide or disable fields on a page.
  • Restrict the valid values for a field using separators (";") and standard wildcards (that is, you can use "*" and "." to represent any character and one character respectively).
  • Move the Facility field.
  • Hide the main strip of tabs on a page or change the way the elements in the tab are displayed in the page (for example, move a field to a different tab).
  • Rename menu options or hide them.
  • Set your chosen value as the default value for an analysis code field in Quantum. Each new deal then has this default value in place, and the default persists even when you change the instrument. You can lock the default against editing and the field appears without a box.
  • Use includedValue or excludedValue to filter a named analysis code, for example:

<AnalysisCode name="Named Analysis Code" includedValue-"e">

In the analysis code field called "Named Analysis Code", the Select dialog box will only show the options starting with "e" or "E".

  • If you select a custom table in Core Module when setting up a Customization page, the FIelds in the XML area are limited to these attributes: id, title. disabled, hidden, defaultValue, includedValues, excludedValues, and source.

Dealing Page Customizations

In addition to the customizations described above... You can create a Dealing page in the Enterprise Workflow (EW) by opening Configuration > Page. The page is based on a particular core module that you select, for example, based on a "Actual Cash Flow" Dealing page. You can then modify this new Dealing page by removing fields, changing captions, and moving analysis codes so that they are in between regular fields. Modifying the new page does not alter the original core module.

You can also assign default values to analysis codes. These default values will override those in any matching Analysis Code Defaulting Rules.

The following sections describe useful information that applies to creating a new Dealing page. For more information about the areas you'll see when configuring the page, see Areas on the Workflow Page.

Variants

You can configure Dealing page variants to share data with their associated core page, and vice versa.

  • Select Include core data to include, in the variant page, deals that are created in the core page.
  • Select Publish data to core to include, in the core page, deals that are created by the variant page.

Validations grid

You can use the Validations area to validate the process of setting up a new deal. In this example we create a validation message that will appear if anyone tries to set up a Dealing page with the bank "Southern".

Example

  1. In Configuration > Page, create your specific workflow page for the deal you are interested in.
  2. In the Validations grid > New Validation dialog box, add one validation with:

    Condition: {[Bank]} ="Southern"

    Message: You cannot select {[Bank]} bank.

    Type: Block; On Creation

  3. Finish setting up the page and save.
  4. Open the new Dealing page and enter Southern as the bank name.
  5. When you try to create a deal with Southern as the bank, the system will not create the deal and the message appears:

    "You cannot select Southern bank."

  6. The system logs the message.

Workflow Actions

Configure workflow actions, associated with a deal you are working on, to update, delete, or create other deals.

For more detail, see Workflow Pages - Workflow Grid.

Analysis Codes

Analysis Codes can be moved away from the Analysis Codes tab and positioned between regular fields. When all Analysis codes have been moved or hidden, the Analysis Codes tab is also hidden.

Variants and Module Security

Module security applies to workflow pages created in the Enterprise Workflow (EW).

When you create a variant page, the system creates a new module. Access to the variant page is controlled through module security on the new module.

NOTE: The variant module has the same set of permissions as the core module that it is associated with.

You can configure module security against the variant module for users and groups just like other modules.

See also: Module Security in Workflow Reports (EW).

Importing and Exporting Customizations

You can export workflow pages into Excel, CSV, XML, PDF, or QXT format from the browse panel of the page. QXT files created this way may then imported into another Quantum site via the Static Data Import Status page.
