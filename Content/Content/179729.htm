EFT Setup - Configuration pages

NOTE: Default perspective only.

SWIFT Local Authentication

(Optional for SWIFT or SWIFT-like EFT output.)

To apply SWIFT Local Authentication to SWIFT EFT output*

  1. On the SWIFT Local Authentication page, select Enable SWIFT local authentication.
  2. Select the Algorithm Type: SA-2 Algorithm or HMAC-SHA256 Algorithm.
  3. Enter the Key's First Part and Second Part. This may require two different users.

NOTE: The system will also apply this authentication to any SWIFT and Finastra (Misys) Electronic Confirmation output.

Users and Module Security

Module Security rules apply as normal to users of the required application pages, as specified on the User page.

Users and Resource Security

If Enable Resource Security is selected for your site (on the Data Security page), the system applies Resource Security rules to data available to users when accessing the required application pages (as normal).

HOWEVER, if Apply to EFT is also selected, the system also applies the user's Resource Security rules to EFT message content.

External Method Source

An External Method Source record must be present for each type of EFT output, as follows:

For SWIFT

  • Name must be set to SWIFT,
  • Type Export may be set to Create File or Queue (see EFT Message Output). Publish to File and Publish to ECHOS removed or read only from 2020.2.
  • Output File Name should be set to *.SFT (note this default may be overwritten on the EFT Processing page),
  • Destination (if available) should be set to an appropriate location (see EFT Message Output; also note this default may be overwritten on the EFT Processing page),
  • Transformation Configuration File is set to the supplied XTC Configuration XML Data record name SeedXMLConfig_EFT (or a modified version of it), OR may be left blank if no XML transformation is required (for example if you are using ATS only),
  • Non-Financial Transfer is set to MT101, MT103, or MT103+,
  • other settings are made as required, and (optionally)
  • IF different SWIFT message formats apply for different counterparties: Counterparty Mapping is set to the file path for the required mapping files. See Counterparty Mapping Files for SWIFT EFTs.

For TARGET - 2020.3 onwards

From 2020.3, we stop using the TARGET2.txt file and include the addressee details in the EFT XML Framework. You can define an external method for TARGET EFT payments.

For more information about TARGET, read Details of TARGET Addressee.

For Other EFT Types (other External Methods)

  • Name is set to a unique value, meaningful for the EFT type,
  • Transformation Configuration File is set to the name of the specially provided or created XML Template Configuration file for the EFT type (see XML Template Files for other External Methods), OR may be left blank if XML transformation is NOT required by Quantum Six (for example if you are using ATS only),
  • CRC Program - an optional name for the cyclic redundancy checking program, if required, and
  • other settings as required.

NOTE: If XML transformation is required for other External Methods, they will need their own specific set of XML Data records and - in some cases - development of special EFT Plug-Ins.

To generate ISO 20022 messages

  1. Run Database maintenance to get the seeded pacs.008_FIToFICustomerCreditTransfer and pacs.009_FinancialInstitutionCreditTransfer workflow templates. These then appear in the Browse list for a Workflow Template page.

  2. Run script SEED_CUSTOM_TABLES.qms to make the seeded Mapping Table page appear in Configuration menu.

  3. Enable Module Security for Workflow Template pages and Mapping Table pages.

  4. Specify the workflow template's name in the relevant External Method page.

  5. Set up the External Method page's Source. Clear the Transformation Configuration File.

  6. Publish the workflow template prior to running EFT processing.

Before using, please check whether batching, File per Day and Single file Efts are supported.

To validate ISO 20022 messages

  1. Save the template with the corresponding ISO 20022 published schema.

  2. The EFT process displays an error when the message is invalid. When this happens, the schema validation message is logged and the EFT output is not created.
