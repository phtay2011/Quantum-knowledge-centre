About Quantum Rates

Quantum requires rates for a variety of purposes that include pricing, marking to market, and currency translation. For example:

  • Position Query applies rates when revaluing the deals specified.
  • Currency Translation applies rates when calculating the currency translation gains or losses on non-FX deals.
  • Deal and Settlement Approvals applies rates if required to convert deal and cash flow currencies for matching with Approval Control records.
  • Unit Prices (in AG-Equities) are created as a type of Rate record for a specific Security Identifier, Date, and Price Type (Rate Type).

Individual rate records accumulate in the database as they are saved.

NOTE: Only one record can be saved per day for each combination of Rate Name, Rate Type, and Currency. For Unit Prices, the required unique combination is different (namely, Security Identifier, Price Date, and Price Type).

Quantum uses the stored records to display historical rates. To calculate risk and exposure positions for an earlier time period, it uses the rates that were current at that time.

When revaluing deals by Instrument, you define which rate is to be applied by specifying the:

  • Revaluation methodology
  • Name of the rate
  • Type of rate

To accommodate the different requirements of Management and Accounting reporting, set up control records using Management Reporting Control and Accounting Rules.

NOTE: You can change the Rate Type displayed before running a query (via the Reporting or Processing menu). It will be changed only for that Query.

To add a new rate name and type, see Rate Name and Rate Type.

Finding the required Rate record

Each individual rate record in the database is identified by a unique combination of:

  • Rate Name
  • Rate Type
  • Currency
  • Date

The system uses this combination to find the required record for a task, which is why identical records cannot be entered for a single date.

For example, when you run Position Reporting or Processing, the system looks in the database for the one rate record that has the date specified for the query (the As At date), combined with the rate name, rate type, and currency of the deals that are to be revalued. This is the record it uses for the revaluation. If there is no rate record for the exact date, the system uses the record with the closest date prior to that, but which otherwise meets the same criteria.

NOTES: Once a record has been selected in this way, all rates for the revaluation are drawn from that record. If a required rate is missing, the system interpolates from the other rates in that record (it does not search for other records containing the missing rate).

If entering new rates by Instrument Type (see Issuer/Instrument Rates), make sure that there is a rate for every instrument that is to be revalued, not just for those where the rate has changed. To achieve this, you can copy an existing rate record that contains all of the instruments, and change the copied version as required. See Copying rate records. The system cannot interpolate between dissimilar instruments.
