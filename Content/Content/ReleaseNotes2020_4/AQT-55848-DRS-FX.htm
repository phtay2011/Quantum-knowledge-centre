Solutions - Regulatory reporting workflow for DRS FX

We have added templates that the system uses to construct the different types of files that the repositories support. In this release these cover reporting on the creation and modification of FX deals.

Run the RegulatoryReporting.QMS file to set up the regulatory reporting functionality setup in the system. In summary, this makes the following available:

  • regulatory workflow templates for EMIR REGIS and EMIR DTCC FX reporting,

  • Regulatory Entity and Regulatory Counterparty workflow pages where the user enters in the regulatory reporting data,

  • the "Report Deal Action" workflow report,

  • a workflow action that processes the selected deal actions by executing the templates to create files, and

  • parsers and Data Transfer Locations (DTL) for processing the incoming response files from the trade repositories.

New pages, workflow templates, and parsers

The Regulatory Reporting rules page defines the jurisdiction to which you need to report for the specific Entity, Counterparty and Instrument. These rules will be used by the "Reportable Deal Actions" data source.

At the same time as the system sets up the source, it creates the workflow templates, "EMIR Regis TR" and "EMIR-DTCC", to use with a DTL and a workflow report. The templates are Read Only.

The system also creates the following pages in the Static menu to store the regulatory reporting account information: Regulatory Entity and Regulatory Counterparty.

For more information, search for the Knowledge Center topic "Regulatory Counterparty".

For more information, search for the Knowledge Center topic "Regulatory Entity".

TIP: The solution is flexible enough to accommodate post-Brexit reporting scenarios. For example, if you add the FCA template to the system and configure correctly, FCA reporting will work in the same way as EMIR.

As well as the rules and static data, the system also creates:

  • an action to update a workflow table that stores the status of current regulatory reporting, and to generate related template output.

  • a new workflow table, named Regulatory Reporting Status, found in the Reporting menu.

  • a new field, named UTI, on the FX Dealing page.

  • a field UTI Prefix on the Regulatory Entity page to be used with trades in regions that need a prefix and an identifier.

  • a new column "UTI Prefix" in the Regulatory Reporting Status page.

  • a new function "IsFXSpot". The function appears in Expression Builder.

  • a new function "CurrencyPair". The function appears in Expression Builder.

  • a new input "Continue On Error" available in workflow template action inputs. Default = No. If No, the system stops execution for the entire data passed to the action and returns an error. If Yes, the system skips the records that are in error and continues the processing for the rest of the items. workflow templates that are of Type = Dialog do not use "Continue On Error".

We altered the fields Instrument and Transaction Type in a source to now use the option List. This allows you to use a "One Of" operator with them in Expression Builder.

About Unique Transaction Identifiers

Unique Transaction Identifier (UTI) is a non-financial field that contains a maximum of 60 characters which must not include comma, period, hyphen or underscore characters

The system maps the deal's UTI to the UTI field values in the Report Deal Actions page.

The value of UTI is used in the regulatory reporting of the trade to EMIR REGIS-TR or EMIR DTCC. If you don't enter a UTI, the system auto-generates a UTI at the time of reporting the trade.

EMIR REGIS-TR

Parsers and DTLs are configured to handle the response files from EMIR REGIS-TR.

EMIR DTCC GTR

Similarly, there is a workflow parser page for processing CSV responses from EMIR DTCC GTR.

In case of delegated reporting, the system considers both ASK messages while processing the response.

NOTE: The EMIR DTCC response files may have two rows for the same deal in the case of delegated reporting: one row for each reporting participant.
