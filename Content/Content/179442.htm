Implementing Confirmations - Summary

System Services

The Process Confirmations service must be running on your Web server to deliver ALL confirmation output.

The Match Confirmations service is required for processing only Electronic confirmations. It must be running on your Web server to receive and automatically match responses from External Matching systems (SWIFT and/or Finastra*) to Electronic confirmation messages sent by Quantum Six.

* Formerly known as Misys.

Prerequisites

Type

Requirements

Site Licensing

Must include Confirmations

XML Data records

Mandatory for Electronic confirmations, and:

  • available via the XML Data page,
  • Seed records are provided for SWIFT and Finastra/Misys*, and
  • users can modify or create new records for new methods.

*NOTE: The SeedXMLDataTemp_ALL record is also provided, mainly for use with the less restrictive Electronic External Method (or optionally with SWIFT or Misys). To use it (or an edited version of it), you need special XML Configuration and Layout records (not supplied). Contact your FIS Consultant for help in creating these.

Crystal Reports

Mandatory for Paper confirmations, and:

  • located in Report Path location configured for site,
  • sample report files are provided,
  • users can modify or create new reports, and
  • Reports must be mapped to the CONFFMTP module (see Crystal Mapping below).

External Matching Service

Mandatory for Electronic confirmations. Your organization must arrange User Registration and the data communications link directly with the Matching service.

Quantum Data Setup

Application Page

Detail

License

Under Back Office:

  • Confirmations must be selected.

Operational Settings

Under Active Services:

  • Process Confirmations must be selected in all cases, and
  • Match Confirmations must be selected for Electronic confirmations.

SWIFT Local Authentication

(Optional for Electronic Confirmations.)

To apply SWIFT Local Authentication to SWIFT and Finastra (Misys) Confirmation output messages (and SWIFT EFT messages):

  1. Select Enable SA-2 message authentication.
  2. Select the Algorithm Type: SA-2 or HMAC-SHA256.
  3. Enter the Key's First Part and Second Part.

User

A ServiceUser record must be present to run the system services (see below), with:

  • Service Account selected, and
  • exemption from Module Security and Resource Security.

(Standard Module Security and Resource Security rules apply to users of the required application pages, as normal.)

External Method

For each electronic matching system used a record must be present with Method Type set to Confirmation and named:

  • SWIFT Confirmation (short name SWIFTCONFO),
  • Misys, or
  • Electronic.

For SWIFT only, a record named SWIFT Code must also be present, with Method Type set to Mapping and Enable Identifiers selected. See also Location below.

Crystal Mapping

All Crystal reports for use with Paper Confirmations must be mapped to the CONFFMTP module.

Paper Confirmation Format Rules

Records must cover all required Matching Criteria (Transaction Type, Entity, Relationship, Counterparty, Instruments) and specify the Default Paper Formats (Crystal reports) for each required Deal action.

Electronic Confirmation Format Rules

Records must cover all required Matching Criteria (Transaction Type, Entity, Relationship, Counterparty, Instruments) and specify the Default Electronic Formats (XTC Configuration XML Data records) for each required Deal action.

Confirmation Rules

Records must cover each required Entity (or ALL), with record entries covering required Matching Criteria (Transaction Type, Instrument, Action, Category, Relationship, Counterparty, Dealer) and specify Settings as required - in particular for:

  • Electronic Confirmations: External Method, Output Subfolder, Electronic File Extension, and
  • Paper Confirmations: Print/Email/File/None, File Subfolder, File Type (see TIP below).

Analysis Codes

For SWIFT Electronic Confirmations, specific Analysis Codes are required with deal actions for:

  • Exotic Currency Options,
  • Interest Rate Swaps, and
  • Cross Currency Swaps.

Location

For SWIFT Electronic Confirmations, the SWIFT Code Identifier field (set up in External Methods - see above) is required for each location involved in Swap deals, populated with a valid SWIFT Financial Center code.

Entity

For each Entity involved in deals requiring Electronic Confirmation, an Identifier named either SWIFT or Misys is required, populated with an individual value (normally allocated by the matching service).

Counterparty

For each Counterparty involved in deals requiring Electronic Confirmation, an Identifier named either SWIFT or Misys is required, populated with an individual value (normally allocated by the matching service).

Counterparty Contact Rules

For each Counterparty involved in deals requiring Paper Confirmation (including as backup to Electronic Confirmation), records must cover the required Matching Criteria (Transaction Type, Entity, Instrument, Currency, and so on) and specify Settings as required - in particular contact names, Address, and SMTP Address.
