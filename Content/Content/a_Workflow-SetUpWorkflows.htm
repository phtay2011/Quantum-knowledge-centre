Components of the DRS

DRS uses the following Event Workflow components:

  • Templates - Jurisdiction specific templates for FX, CO, IRS, CCS, IRO, VU, and CU_CL. The following templates are stored in Quantum and used in the reporting process:

  • Data Transfer Locations For template output, these are the locations which are added as actions on the Workflow Action screen.

  • Workflow Report - The REGREP workflow report is responsible for showing all the trades that need to be reported.

  • Workflow Tables:

    1. Regulatory Reporting Status: A workflow table that holds all the information of the trade reporting life-cycle.

    2. Regulatory Entity: A workflow table screen for holding the static data relevant to trade reporting for any jurisdiction.

    3. Regulatory Counterparty: A workflow table screen for holding the static data relevant to trade reporting for any jurisdiction.

    4. Regulatory Reporting Rule: A Rule type of workflow table for holding rules that will provide information on the reporting jurisdiction, type of reporting, Third Party LEIs for any Entity or Counterparty for reporting.

    5. Mapping: A workflow table for mapping values from Quantum to regulatory specific values. This table is generic and can be used for a variety of modules.

  • Workflow Action: A workflow action named as Process Deal Actions is available. It is a chained action that has the Template action for all the transaction types supported for reporting across the EMIR and FCA regulation. This chained action is also responsible for updating the status table when a file is generated for reporting any deal actions.

  • Workflow Parsers Workflow parsers are configured for processing response for Regis and DTCC repositories for EMIR and FCA regions. The Parsers are : EMIR DTCC Response, EMIR REGIS Response, FCA DTCC Response , FCA REGIS Response, EMIR REGIS SFT Response. The EMIR DTCC Response now supports processing EMIR DTCC Valuation response files.

  • Parser DTLs Parser DTLs defined for the Parsers mentioned earlier. The DTLs have an action that reads the incoming file content and appropriately creates a row in the regulatory reporting status table to reflect the correct status of reporting.

Static Data Setup

The following static data elements are part of the Regulatory Reporting process:

  • Regulatory Entity - Add jurisdiction and repository specific data applicable to the entity.

  • Regulatory Counterparty - Add jurisdiction and repository specific data applicable to the counterparty.

  • Regulatory Reporting Rule - This is a rule-type of a workflow table. Use it to define rules that you join with the base source on certain columns in the report so that the report returns the correct trades for processing.

You can define a rule to select trades with matching Entity, Counterparty and Instrument. The rules are effective from the Effective Date. This means that the best match rule would also be the latest rule as per the date value used in the setting of this workflow table source.

NOTE: If you don't specify the effective date, the system will take the current date for comparison and return the best matched rule.

  • Collateral Type

We provide support for collateral tags in the workflow templates, for example, in the workflow templates "EMIR Regis-TR" and "EMIR DTCC". This functionality allows you to report the compulsory Collateral tags with a new trade submission.

  1. To use this on deals, in Configuration > Analysis Codes create an analysis code named "Collateral Type".
  2. In the Predefined Values grid you can add "Fully" (FC),"OneWay" (OC), "Partially" (PC), or "Uncollateralized" (UC).
  3. If you see "U" in the output, this means "Not Set".
