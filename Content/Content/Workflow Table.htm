Table

The Table component provides a way to maintain solution specific data. Creating a workflow table results in:

  • a page to maintain the data,

  • a data source to retrieve the data from the table for other workflow components to use, and

  • workflow actions to create, update, and delete data from the table.

How to set up a table

Workflow tables can be created by importing a JSON file that contains the table definition. When the JSON file is imported, one or more workflow tables are created depending on whether a detail table is used and approval is enabled. The full table definition is stored in the database.

TIP: You need to perform an IISRESET after the import for the edit pages to take effect.

Create table example

The below example creates a Workflow Table with four columns (Name, Counterparty Identifier, Domestic Currency, and Non-European).

Copy 
[{
    "application": "WorkflowTable",
    "action": "Create",
    "identifier": "CPINFO",
    "data": {
        "name": "CPINFO",
        "title": "Counterparty Info",
        "tableName": "counterparty_info",
        "urlName": "counterparty-info",
        "menuArea": "Static",
        "menuGroup": "Other",
        "allowDeactivation": true,
        "approveChanges": true,
        "levelsOfApproval": 2,
        "approvalRole": "cparty_approver",
        "tableColumns": [{
            "name": "name",
            "title": "Name",
            "key": true,
            "reference": "cparty"
        },
        {
            "name": "counterparty_id",
            "title": "Counterparty Identifier",
            "dataType": "Text",
            "size": 60,
            "inGrid": true
        },
        {
            "name": "domestic_ccy",
            "title": "Domestic Currency",
            "reference": "currency",
            "inGrid": true
        },
        {
            "name": "non_european",
            "title": "Non-European",
            "dataType": "Yes/No"
        }]
    }
}]

The Name is a lookup on the cparty table and the Domestic Currency column is a lookup on the currency table. The Name column is a key field and will therefore appear in the browse grid of the edit page as well as Counterparty Identifier and Domestic Currency. The edit page sits in the site menu under Static > Other. The data in the table is under approval and requires two approvals from users that have the Counterparty Approver role.

Delete table example

The below example deletes a Workflow Table through JSON import (without executing database maintenance on the server).

Copy 
[{
    "application": "WorkflowTable",
    "action": "Delete",
    "identifier": "CPINFO"
}]

Detail tables

A workflow table can be defined to contain a detail table which will create a one-to-many relationship between the main table (the header) and the detail table. The detail table has a different set of columns.

Detail tables allow for a more efficient storage of data (compared to a single table) and will display the related detail rows with the header row while editing the data.

Viewing table configuration

A Workflow Table page is available to view the configuration of the workflow tables set up in the database. This page only shows workflow tables created through JSON importation. Workflow tables created using scripts will not appear.

NOTE: The previous method of creating workflow tables (using scripts executed by DatabaseMaintenance) will be discontinued in a future release and will be blocked by DatabaseMaintenance from 2021.4.

The workflow table's browse list menu includes an Export option which allows you to export these workflow table configurations into JSON.

Using tables

Edit table data

Each workflow table comes with a page that allows editing its data.

The data in a workflow table can be retrieved through a data source for use by other workflow components (e.g. reports, actions, or pages). If a workflow table contains a detail table, the data source combines the data of both the header and detail table.

Editing rights are controlled by module security.

Retrieve data from a data source

The data in a workflow table can be retrieved through a data source for use by other workflow components (e.g. reports, actions, or pages). If a workflow table contains a detail table, the data source combines the data of both the header and detail table.

Manipulate data through workflow actions

All workflow tables come with workflow actions to create, update and delete the data. If a workflow table consist of a header table and a detail table, then there will still be one set of create, update and delete actions. All columns of both header and detail tables will be represented as inputs to the actions.

Approve data

When approveChanges is set to true, all changes made to the data in the table will go through Static Data approval. The levelsOfApproval setting determines how many approvers need to approve the change (between 1 and 99) and the approvalRole setting sets which type of Static Data approver can approve the data.

Match rules

Match rules

Workflow tables can be set up to be treated as rules. A rules table comes with an additional data source that fetches the best matching rule using logic similar to standard Quantum defaulting rules. Instead of just returning exact matches, as is the case for a regular workflow table data source, the rules data source returns a match on group or on ALL.

To set up a workflow table to store rules, first set useAsRules to true. Then the following applies for the Rules data source:

  • A setting is available for each of the key fields in the header and detail tables.

  • The settings are used to find the best matching rule.

  • Only the best matched rule is returned as a single row with all columns.

  • When no rule matches, the data source does not return any rows.

  • When finding the best match, the following logic applies:

  • exact match over group match,

  • group match over ALL,

  • field priority from left to right, and

  • deactivated rules are ignored.

Set effective date

When a workflow table contains an Effective Date column, then this date is used to retrieve the most appropriate and recent row from the table that matches the effective date.

To set up the workflow table to use effective dates, include an Effective Date column in the table, and the data source will show one additional setting, called As At.

The following applies:

  • The Effective Date column name is case insensitive and the space can either be removed or replaced with an underscore.

  • If a workflow table has an Effective Date column, then an As At setting is available on the data source.

  • When As At is set, the data source returns the appropriate data from the table, that matches the effective date.

  • When As At is not set, the data source returns the most recent (to today).

Upgrading workflow tables

 

WARNING:The previous method of creating workflow tables (using scripts executed by DatabaseMaintenance) will be discontinued in a future release and creation of new workflow tables will be blocked by DatabaseMaintenance from 2021.4.

Workflow tables defined using scripts will need to be manually upgraded to the new JSON syntax. To assist with this process a solution is available that generates JSON files for existing workflow tables.

Install upgrade solution

The solution consists of the following components, which you need to create in the following order:

  • database view qvw_datasource_workflow_table_upgrade

  • template Convert Table to JSON,

  • report EWT_UPG , and

  • mapping Data Type.

Download existing data

If data in the existing workflow tables need to be kept, export the data from the tables. Use the CSV format.

Generate JSON file

  • Open the Upgrade Workflow Tables report.

  • Select the table that needs converting.

  • Run query.

  • Select all rows.

  • Optionally, use Convert Table to JSON... to preview the JSON

  • structure or observe any errors.

  • Use Download JSON to download the JSON file.

Import JSON file

From an account that has change permission on the WorkflowTable module import the JSON file.

The JSON file includes a Delete action which will remove the old table from the system and a Create action which will add the table back in using the new syntax.

TIP: Perform an IISRESET after the import for the edit pages to take effect.

 

NOTE:The names of the primary and foreign key columns are no longer userdefinable and are generated by the system. Review your logic if your expressions referred to any of these columns.

Import data

  • Using the exported CSV file, set up a parser.

  • Define a DTL that uses the parser and calls the Create action for the workflow table.

  • Import the CSV file through the DTL.
