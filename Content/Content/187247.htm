Replicating FX Requests into Analytics

Quantum FX Requests are replicated into Analytics for pricing and reporting purposes, including (but not limited to):

  • Request vs Actual deals at various points in time,
  • which Requests have been filled over a period including type and volume,
  • outstanding Requests,
  • breaking Request volumes down by region and entity, and
  • aggregating Requests by key properties - for example, Currency, Entity, and so on.

FX Requests are replicated to Analytics by default, unless the BLOCKREQUESTS custom setting is set to YES for the Analytics server. See the Installation Instructions for more information.
Once an FX request has been filled then the resultant FX deal should flow from Quantum to Analytics, regardless of whether you set BLOCKREQUESTS to YES or NO.

Analytics Instruments

Quantum FX Instruments with Type set to OUTRIGHT or SPOT & FORWARD have an Allow requests option. Such Instruments will be replicated into Analytics as FX Forward Instruments that display the applicable Allow requests option setting in read-only mode.

TIP: The Allow requests option is not applicable to FX Par Forward Instruments.

Analytics Pricing

Quantum FX Deal Requests will be replicated into Analytics as FX Forward Deals with their Deal Type set to Request. They are viewable on the FX Forward Pricing page*, with Instrument set to the replicated Quantum FX Instrument and the read-only Deal Type field showing Request. If required, the matching Deal Request Rule record will be used to populate missing field values required in Analytics (see Deal Request Defaulting Rules below).

While FX Requests cannot be saved, edited, changed, or filled in Analytics, you can use the Copy for Calculation function to create a new, editable FX Forward deal with Deal Type set to What If (in line with previous functionality), with properties on the Deal tab set to blank or zero and all the functionality of a standard What If deal.

Any editing or deletion of an existing FX Request in Quantum will flow through the Replication process and update that FX Request in Analytics.

When an FX Deal Request is filled in Quantum (either by creating a new FX Deal linked to the Request or by linking an existing FX Deal to the Request), the results will be replicated in Analytics as follows:

  • the FX Request will be linked to the FX Deal at a database level and via a visible tag Filled by Dealid (where Dealid is the FX Deal's ID),
  • the linked FX Deal details will be replicated (created if a new deal, updated if an existing one), and
  • the linked FX Deal will be linked to the FX Request at a database level and via a visible tag Fills request Reqid (where Reqid is the FX Request's ID).

When a filled FX Request or filling FX Deal is edited or deleted in Quantum, the results are also replicated in Analytics as follows.

  • When a filled FX Request is edited using the Reset menu function, the link between the FX Request and the FX Deal will be cleared and no linkage will remain at the database level.
  • When a filled FX Request is deleted using the Delete menu function, the FX Request will be tagged as Deleted, the link across both the FX Deal and FX Request will be cleared, and no linkage will remain at the database level on either deal.
  • When a filling FX Deal is edited using the Reset menu function, the FX Deal will be updated but the linkage to the FX Request will remain in place.
  • When a filling FX Deal is deleted, the FX Deal will be tagged as Deleted, the link across both the FX Request and FX Deal will be cleared, and no linkage will remain at the database level on either deal.

*TIP: Unlike in Quantum, there is not a separate Pricing page for FX Requests in Analytics.

Analytics Deal Request Defaulting Rules

The Analytics Deal Request Rules page is accessible via the Analytics perspective's UTILITY menu. Use this page to create and maintain defaulting Rule records that will be used by the Replication process to supply any missing field values that, while required in Analytics Pricing, are optional in Quantum Requests.

The applicable fields are Counterparty, Entity, Dealer, Spot Rate, Contract Rate, and Forward Points.

Where such values are missing from FX Deal Requests replicated into Analytics, the Replication process will use the closest-matching FX Deal Request Rule record to supply the applicable values.

Each Rule record contains two sets of lookup fields, as follows.

  • Matching Criteria:
    • Instrument - select an FX Forward Instrument, Instrument Group, or ALL;
    • Buy Currency - select a Currency, Currency Group, or ALL; and
    • Sell Currency - select a Currency, Currency Group, or ALL.
  • Settings:
    • Entity - select a specific Entity or Request Entity*,
    • Counterparty - select a specific Counterparty or Request Counterparty*, and
    • Dealer - select a specific Dealer or Request Dealer*.

*TIP: The default values (Request Entity, Request Counterparty, and Request Dealer) are provided as seed data in the Analytics database.

A seed or "cover-all" rule is provided so that the system can find at least one rule record to cover every replicated request. It is recommended that you DO NOT delete that entry, which has the following settings:

  • Instrument is ALL,
  • Currency is ALL,
  • Entity is Request Entity,
  • Counterparty is Request Counterparty, and
  • Dealer is Request Dealer.

Analytics Deal Analyze

In Analytics Deal Analyze, an FX Request can be retrieved via the Browse button or by entering the Analytics FX Forward deal number or deal identifier for the replicated Quantum Request.

The deal request can be valued using the Calculate button, as for any other FX Forward deal.

TIPS:

  • You can filter the browse list by Type to include (or exclude) Requests.
  • Where a Rate has not been specified on the deal, an error message is displayed stating that the system cannot value the deal without a Rate.

Analytics Reporting

An FX request entered inQuantum and replicated into Analytics will not have any Bank Accounts associated with the Buy or Sell amounts in the schedule. Therefore Analytics Position, Cash Flow, and VaR reports with an incorporated Balance definition (type: Bank Balance or Entity) will not have their balances impacted by any FX Deal Requests.

Across all report types:

  • the new Allow Request setting in FX instruments is available as a Database Column through the Instrument folder and can be output to a report,
  • the new Request deal type in FX Requests can be output through the existing Deal Type Database Column,
  • the Request Counterparty, Request Entity, and Request Dealer default settings that can be specified during the Replication process (from Deal Request Defaulting Rules) are available via the respective Group By columns,
  • all other FX Request information can be reported on as per any standard FX Forward deal,
  • the Query Filter Definition allows Requests to be included or excluded from reports via the Deal Type filter,
  • where an FX Request has an amount specified for only one leg, only that leg will be output in the report (see Note below),
  • where an FX Request has been filled inQuantum, the linked FX Deal Id, date of linkage, and type of linkage can be extracted through the Database Columns and output, and
  • for an FX Deal filling a Request, the linked FX Request Deal Id, date of linkage, and type of linkage can be extracted through the Database Columns and output.

An FX Request entered inQuantum may specify a Buy OR Sell Amount only, and no Contract Rate. When such an FX Request is replicated into Analytics, the system cannot price the deal in the Pricing screen nor in Deal Analyze, and Position and Cash Flow reports can only report on the deal leg that has an amount specified.

Exclusions and Conditions

FX Requests:

  • cannot be entered directly into Analytics via the Pricing page, Quantum Deal Import page, or Scheduler Deal Import Task Provider, and
  • cannot be filled in Analytics.
