message element, Data Template XML

Confirmations only.

Only one message element is allowed per Confirmation element.

Structure:

<message/>
  <currentMessage/>
  <previousMessages>
    <previousMessage/>
  </previousMessages>
</message>

currentMessage Element, Data Template File

Confirmations only.

The currentMessage element deals with providing the necessary information to construct the current message such as Message Ids, and so on.

Only one currentMessage element is allowed per message element.

Structure:

<currentMessage/>
  <messageID source="..."/>
  <externalReference source="..."/>
</currentMessage>

For more details, see the SeedXMLDataTemp_ALL XML Data record provided in your Quantum database.

previousMessages Element, Data Template XML

Confirmations only.

The previousMessages element provides any details about the last message (if any) associated with the current deal.

Only one previousMessages element is allowed per message element.

Structure:

<previousMessages>
  <previousMessage/>
</previousMessages>

For Confirmations, the linkage to a previous message is explained in the following previousMessage section.

previousMessage Element, XML Data Template

Confirmations only.

Function

Finds the message ID of any previous messages linked to the current Action. Once found, the content structure and sources defined in the SeedXMLDataTemp_ALL seed record are used to to populate the previousMessages element for each message found, in descending message order (that is, the latest message will be the first).

Only one previousMessage element is allowed per previousMessages element.

Structure

<previousMessage/>

  <messageID source="..."/>
  <messageType source="..., ..."/>
  <messageActionType source="..., ..., ..."/>
  <objectType source="..., ..."/>
  <externalReference source="..."/>
  <messageContent source="..."/>
  <otherInfo source="..., ...">

    <cpartyReference source="..."/>
    <matchMethod source="..."/>

  </otherInfo>
  <lifeCycle source="..., ...">

    <lifeCycle source="..., ...">

      <lifeCycleID source="..."/>
      <lifeCycleState source="..."/>
      <lifeCycleDescription source="..."/>

    </lifeCycle>
    <subLifeCycle source="..."

      <lifeCycleID source="..."/>
      <lifeCycleState source="..."/>
      <lifeCycleDescription source="..."/>

    </subLifeCycle>

  </lifeCycle>

</previousMessage>

For more details, see the SeedXMLDataTemp_ALL.xml file provided in your Quantum Site's Confos folder.

Rules - DEAL Action

There will be no previous message for any DEAL Action.

All Transaction types have a DEAL Action, and will not have had a previous message sent at this stage.

Rules - CHANGE

All Transaction types have a CHANGE Action and will need to be checked for any previous messages.

There can be multiple CHANGE Actions for a deal. Each of these CHANGE Actions may or may not relate to a message previously sent.

For the current CHANGE Action, you will need to look for the last CHANGE or DEAL Action that had a message sent.

(Currently, there is no requirement to show multiple previous messages.)

Rules - DELETE

All Transaction types can have a DELETE Action and will need to be checked for any previous messages.

For the current DELETE Action, you will need to look for the last CHANGE or DEAL Action that had a message sent.

(Currently, there is no requirement to show multiple previous messages.)

Rules - RATE FIX

Transaction Types Affected

FR - Forward Rate Agreement (FRA), FX - Foreign Exchange deal - NDF (Non Deliverable Forward) only

Requirement

There can be only one RATE FIX Action for a deal. Each RATE FIX Action relates to a message previously sent.

RATE FIX Actions occur through the individual Deal program's Rate Fix form (and its QTOM equivalent) only.

Rules - RATE SET

Transaction Types Affected

SI - Interest Rate Swap, SC - Cross Currency Swap

Requirement

There can be multiple RATE SET Actions for the same RATE EVENT of a deal. Each of these RATE SET Actions may or may not relate to a message previously sent.

RATESET Actions occur through the individual Deal program's Rate Set form (and its API equivalent) or through the Bulk Renegotiation program.

The Deal program's Rate Set form:

  • will Action any Rate Set through to the current Rate Set period, if they are back dated Rate Sets,
  • has a default to automatically use the same rate for the future Rate Sets - this needs to be specifically turned off, and
  • allows the re-saving of a Rate Set.

Bulk Renegotiation Rate Set:

  • will only Action a Rate Set that fulfills the parameter criteria, and that also remains selected as an item in the selection tab,.
  • will only Action any previously dated Rate Set(s) to that of the current Rate Set that has not already been Actioned,
  • does not allow the re-saving of a Rate Set, and
  • will Rate Set both sides of a Swap if the Rate Set parameters (in Bulk Renegotiation) are the same.

Actioning of a Rate Set will create a new Action called RATE SET and stamp this action number on the RATE SET Event.

When re-saving a RATE SET (from a Deal program's Rate Set form):

  • This will also create another Action called RATE SET.
  • The new RATE SET Action number will be stamped on the RATE SET Event, and any previous RATE SET Action linked to the same Event will have to be obtained by using the Imaged Events.
  • This means that for a RATE SET Event, there have been many re-saves of a RATE SET, which can only be found through looking at the Imaged Events. That is, you need to see the structure of the deal as it stood when that side of the Swap was Rate Set.
  • For the current RATE SET Action, you need to look for the last RATE SET Action that had a message sent.

(Currently there is no requirement to show multiple previous messages.)
