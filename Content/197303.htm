Implementing EFTs - Summary

System Services

[Not applicable.]

Prerequisites

Type

Requirements

Site Licensing

Must include EFTs.

XML Data records

Mandatory if using XML transformation* for EFTs, and:

  • available via the XML Data page,
  • Seed records are provided for the SWIFT External Methods, and
  • users can modify or create new sets of seed records for new methods.

*NOTE: XML transformation in Quantum is optional. It is not required, for example, if you are transforming EFT output using only ATS.

EFT Plug-Ins

A default EFT plug-in is provided that handles SWIFT External Methods. For others, you may require specific plug ins.

Crystal Reports

Optional for EFTs, and:

  • located in Report Path location configured for site,
  • sample report files are provided,
  • users can modify or create new reports, and
  • Reports must be mapped to the EFTRPT module (see Crystal Mapping below).

External Payments Service

Mandatory for EFTs. Your organization must arrange User Registration and the data communications link directly with the payment processing service.

Operational Settings

Not applicable

SWIFT Local Authentication

Optional. To apply SWIFT Local Authentication to SWIFT EFT messages:

  1. Select Enable SA-2 message authentication.
  2. Select the Algorithm Type: SA-2 or HMAC-SHA256.
  3. Enter the Key's First Part and Second Part.

User

A ServiceUser record must be present to run the system services (see below), with:

  • Service Account selected, and
  • exemption from Module Security and Resource Security.

Standard Module Security and Resource Security rules apply to users of the required application pages, as normal. If Apply to EFT is also selected, the system also applies the user's Resource Security rules to EFT message content.

External Method Source

For each payments system used a record must be present with Name set to SWIFT (or a suitable name if you are using new sets of seed files for new methods), with:

  • Transformation Configuration File is set to the required XTC Configuration XML Data record name, and
  • other settings as required - see under EFT Setup - Configuration Pages.

External Method

For each payments system used a record must be present with Method Type set to Payment and named SWIFT (or a suitable name if you are using new sets of seed files for new methods), with:

  • Export Source set to the relevant External Method Source,
  • EFT Plug-In / Template set to SWIFT for those methods, and
  • other Options as required - see under EFT Setup - Static Data and Rules.

For SWIFT only, a record named SWIFT Code must also be present, with Method Type set to Mapping and Enable Identifiers selected. See also Location below.

Crystal Mapping

All Crystal reports for use with EFTs must be mapped to the EFTRPT module.

Location

For SWIFT Electronic Confirmations, the SWIFT Code Identifier field (set up in External Methods - see above) is required for each location involved in Swap deals, populated with a valid SWIFT Financial Center code.

Entity

For each Entity involved in deals requiring EFTs, an Identifier is required with the appropriate name (for example SWIFT, BIC Code, or EDIFACT), populated with an individual value (normally allocated by the relevant authority). See under EFT Setup - Static Data and Rules.

Counterparty

For each Counterparty involved in deals requiring EFTs, an Identifier is required (as for Entity above).

Bank Account

For each account using EFTs, ensure that:

  • the correct Bank Account Defaulting Rule will be applied, and
  • the Allowable Payment Methods include the required EFT External Method.

Counterparty Bank Account

Optional with the SWIFT External Method for each account using EFTs, set Payment Category as required. See under EFT Setup - Static Data and Rules.

Stock Account

Optional with the SWIFT External Method for each account outputting MT540-543 EFT stock flow messages, ensure that:

  • the correct Stock Account Defaulting Rule will be applied, and
  • Allowable SWIFT MT Formats and Default SWIFT MT Format are set to the supplied .QMT file or a customized version. See under EFT Setup - Static Data and Rules.

Bank Account Defaulting Rule

Ensure that records:

  • cover all required Matching Criteria, and
  • have Instructions entries for each Bank Account using EFTs.

See under EFT Setup - Static Data and Rules.

Stock Account Defaulting Rule

Optional with the SWIFT External Method if any MT540-543 EFT stock flow messages are to be output, ensure that records:

  • cover all required Matching Criteria, and
  • have Instructions entries for each Custodian and Stock Account using SWIFT EFTs.

See under EFT Setup - Static Data and Rules.

Running and Monitoring

Quantum page

Use to...

Settlements

View settlement flows and optionally:

  • change the Payment Method on selected flows for one account,
  • reset an Exception flow to EFT Created or EFT Not Created,
  • view EFT message content, or
  • trigger or undo an EFT Cancelation request.

EFTs Processing

Selectively view, monitor, and manually initiate unsent EFT messages for all Payment Methods.

EFTs Reporting

View EFT output for all Payment Methods, and optionally:

  • report on earlier EFT Processing Update runs,
  • view a log of recent EFT Processing and Reprocessing Update runs, and/or
  • create XML Data and/or Debug files.

EFT Messages

View EFT messages output to an external payments processing service.

Create and send EFT Amendment requests.

Free Format MTn99 Messages

View existing free-format messages that have been manually created.

Create free-format text messages based on a selected EFT message.

EFTs Reprocessing

View and optionally reprocess EFT message output when exceptions arise (for example, because of problems communicating with the external payments service).
