How Quantum saves a deal

This section covers saved deal components and other aspects related to saving a deal in Quantum.

NOTE: By default, users cannot change deals that have been fully approved. If you need specific users to have the ability to change fully approved deals, you need to select the override option Allow change of fully approved deals by authorized users only (Data Security page). On the User page, you then need to select the Can change approved deals option (under User Rights) for each specific user.

Standard deal information

You need some information for all types of deal. The standard deal information includes:

  • Deal Number
  • Transaction Type
  • Instrument Types
  • Accounting methodology
  • Deal Dates

Specific deal information

Each type of deal requires some specific information. The information is different for each transaction type and is stored in the deals table for each transaction type. For example, the table MMdeals for Money Market deals or COdeals for Currency Option deals.

This table contains information such as formulae and specific transaction parameters. When you select an instrument type, the system provides default deal pricing and setup parameters from this table. When you save a deal, or prior to this, when you preview general ledger entries and flows, the system applies defaults to determine the following:

  • Bank account information to save cash flows.
  • Custodian account information to save associated stock flows, if any.
  • Counterparty bank information to set flow settlement parameters.
  • Counterparty bank information to set cash flow settlement parameters.
  • Deal approval requirements.
  • Deal confirmation requirements.
  • Deal accounting requirements and chart of account codes to store the General Ledger entries.

Mandatory items

You must add some information before you save the deal:

  • All cash flows must be directed to a bank account, and any associated stock flows must be directed to a Custodian account. So you must set up the bank and custodian account defaults.
  • Accounting control is mandatory because it provides essential book value and revaluation information. The reporting process uses accounting control, even if you don't require accounting at a general ledger level.

Bank Reconciliation Match Integrity enforcement

When you save a deal, Quantum enforces Bank Reconciliation Match Integrity.

Any Match statuses or Reconciliation Marks affected by additional, changed, or deleted flows will be reset or rolled back, to maintain Match Integrity during all phases of the flow life cycle. When a previously Matched flow is reset to Unmatched, all flows that participate in the same match will also be reset at the same time.

The integrity of the Reconciliation mark is maintained. If an account is marked as Reconciled, it cannot have any unmatched flows prior this mark that are not reconciled. This includes bankflows, external cashflows, and cashflows. This means that:

  • If the account was marked as reconciled (by using the Mark As Reconciled function) for a date on or after the Value Date of unmatched bankflows, the Reconciliation Mark is rolled back to the last mark prior to the Value Date of the unmatched items.
  • If a bankflow, cashflow, or external cashflow is created with a Value Date prior to the last Reconciliation mark, the Reconciliation mark is rolled back to the last mark prior to the Value Date of that flow.

Imaging/Auditing

When you change and re-save a deal, the system stores the existing information in a history image.

Access past changes to a deal by viewing Audit History from within the dealing page.

Avoid re-application of Bank Account Defaults on Money Market deals

If the Money Market deal is saved via the API, you can set up your Quantum system to "freeze" the default Bank Account settings before a new Cash Rate is applied via the API interface. This functionality covers Money Market deal cash flows that have the Status ACTIONED and which have had no change to the core Cash Flow attributes - namely Amount, Currency, Settlement Date, Counterparty, Deal Number, Effective Date, Entity, and Instrument.

A flag on the API interface for MMDEALS can be set to Yes or No. If Yes is selected, cash flow information will be frozen and the Bank Account defaults will not be re-applied.
