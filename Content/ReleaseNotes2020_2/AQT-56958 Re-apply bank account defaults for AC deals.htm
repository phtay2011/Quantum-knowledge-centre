Bank account defaulting for Actual Cash Flow deals

The bank account defaulting behavior has been enhanced for Actual Cash Flow (AC) deals.

Background

For defaulting the bank account on AC deals, the system uses Instrument, Counterparty (optional), Entity, Facility, Currency, and Payment/Receipt as matching criteria. If the Counterparty is unassigned, the system uses ALL for the matching criteria for the defaulting rule.

NOTE: The Amount must be non-zero to save a deal.

To determine the Cut-off Time Rule (if any) for an AC deal, the system uses Instrument, Counterparty, Entity, Currency, and Bank Account as matching criteria.

For a new deal created from the AC Deal page, or a variant, or a Workflow action

If you change any of the applicable matching criteria fields prior to saving the deal, the system applies the default bank account fields, and resets the Cut-off Time Rule (if any) accordingly, if required. You may still change to a different bank account before saving the deal, if required. If a Cut-off Time Rule applies, then the system re-evaluates.

For an AC deal created from Cash Explorer or Bank Reconciliation

If you change any of the matching fields, such as Counterparty, the bank Account Number remains unchanged.

Previously, when creating an AC deal from Cash Explorer, such changes could revert the Account Number to the default. For more information about previous functionality, see History .

Importing

Importing of AC deals (for example, by using QXT files) now supports correct bank account defaulting. For example, if the Counterparty is set after the bank account has defaulted, the system re-defaults the bank account.

IMPORTANT NOTE: If a non-default bank account is set in the import file, it must be set after the matching criteria fields to override the defaulted bank account.

Copying a deal to create a new deal

Quantum now ensures that the bank account fields from the source deal are set on the target deal when you copy an AC deal, avoiding any defaulting that may have occurred during the copy process. As the target is a new deal, defaulting of the bank account fields re-occurs if you change any of the matching criteria fields before saving that new deal.

Editing an existing saved deal

Changing any of the matching criteria fields will NOT revert the bank account fields to their default values. This includes deals that existed prior to upgrading Quantum to V20.2.0 from a previous release.

Converting PC to AC

During conversion of a Projected Cash Flow (PC) deal to an AC deal, the bank account fields from the PC deal override any default values on the AC deal's bank account fields.

Exporting

The system now sets Counterparty before other matching criteria for AC deal records in exported QXT files, so if you import the same QXT, this will not result in a different default bank account.

History

Bank account defaulting in AC Deals was introduced with the addition of Cut-off Time Rules in Release 6.8.0. Before that, there was no bank account defaulting.

From Release 6.8.0, bank account defaulting applied during the creation of a new deal (before saving) so that the Cut-off Time Rule could be applied to the Settlement Date, and once the account number had a value, it was not re-defaulted if further changes were made to deal before saving. So, if any of the criteria fields were changed in an existing AC deal, and the account number field was:

  • already assigned - it did not revert to the default, or
  • cleared – it reverted to the default.

Also:

  • AC deal records in exported QXT files had Counterparty set after Entity, and if the same QXT was imported – for example, after upgrading from an earlier release - a different default bank account could result.
  • A zero or blank Amount always applied the Receipt bank account defaulting rule to an unsaved AC deal – that is, it ignored a Payment setting.
  • When an AC deal was created from Cash Explorer, and the Account Number was set from the grid context, any further change, such as changing the Counterparty, could revert the Account Number to the default.
