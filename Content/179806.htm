Supporting elements, Layout XML Data record

This section covers the supporting elements in the Layout record.

Nesting of elements, Layout XML Data record

Currently, all supporting elements described in this section can be nested within each other.

Nesting enables users to construct sophisticated conditions and rules in order to build the correct contents for complex fields.

Structure

A Condition may be specified inside a PriorityGroup, which may be under a MatchingGroup, which may be then under a Combination, which may be then under a PriorityGroup. An example of such a structure follows.

Combination element, Layout XML Data record

The Combination element occurs when the contents of a Field is constructed from multiple FieldItem elements that may or may not have multiple sources of data. During the combining of the FieldItem elements, there maybe a requirement to sort the content, in alphabetical order of each value.

Structure

Rules

A Field or FieldItem element may contain zero or many Combination elements.

A Combination element may contain FieldItem elements or any other nest-able elements.

Example 4 (byPass)

Quantum supports the byPass attribute at a Combination element in the XML Layout File. This means that you don't have to use the PriorityGroup element and that the element can return all values in the combination instead of just the highest priority item in the combination. 

When used, the string produced is the collection of all FieldItem values in the combination.

<Combination byPass="//">
    <FieldItem>//</FieldItem>
    <FieldItem></FieldItem>
</Combination>

The field will be empty.

Example 5 (If xpath is C)

<Combination byPass="//">
    <FieldItem>//</FieldItem>
    <FieldItem xpath="..."/>
</Combination>

If the xpath returns a value "C", then the field is: //C, otherwise it will be empty.

Example 6 (If xpath is no value)

<Combination byPass="//">
    <FieldItem>//</FieldItem>
    <FieldItem xpath="..."/>
    <FieldItem>abc</FieldItem>
</Combination>

If the xpath returns no value, then the Field is: //abc.

NOTE: You don't have to use the PriorityGroup element with bypass.

PriorityGroup element, Layout XML Data record

Based on the Priority level of the value contained by each source within a Field element, only one of them is used.

Structure

Rules

The topmost item has the highest Priority. If it contains a non-empty string, then that value will be used; otherwise, the second item will then be considered, and so on.

If all the items in a PriorityGroup contain either an empty string or the specified byPass value, then the Field element will not have a value, and it can be excluded, or an error message produced.

Condition element, Layout XML Data record

In many situations, an element and its children are only mandatory if some certain Conditions are satisfied. The Condition element is used to specify these conditions.

Structure

Rules

A number of rules must be adhered to in order for the program to recognize the Condition and correctly evaluate it.

Sometimes, one Condition can be referenced multiple times. In order to save time and effort, Conditions can be defined globally and be referenced from multiple places by the combination of ref and id. Inside ref, an ID is used to link to the actual Condition declaration, which is placed at the end of the Layout file.

Attributes

ref and id

These attributes define the linkage between the Condition element and the Conditions definition section.

This attribute can contain:

  • xpath
    • Used to refer to the data source from the Data Template XML Dtata record.
    • Must start with a forward slash (/) and be specified within two # characters - for example:

      "#/confirmation/action/deal/dealNumber#"

    • For more details, see Data Selection (xpath), Layout XML Data record.
  • Element IDs
    • These can be used to refer to a data source from the current file.
    • The id name has to be specified within two # characters.
    • A colon (:) is used as a separator between the element and its parent one layer above.

      For example, if Section has id="C" and Field has id="14F" then "#C:14F#" would obtain the value from that Field. If the colon is omitted, the program will use the first element it finds with the specified id value.

  • Comparison operators: =, >, >=, <, <=, !=
  • String manipulation functions: contains, starts-with, ends-with, string-length
  • Boolean operations: and, or, not
  • Constant strings, placed within single quotation marks - for example "'some value to be compared with'".

NOTE: All the test conditions applied to the current element must be specified as one single string.

MatchingGroup element, Layout XML Data record

Matching occurs when the content of an element is determined by the result from a comparison of multiple sources.

Structure

Rules

Inside MatchingGroup, the AllowableValues element defines all possible values that can be assigned to the element being defined. The value that will be used is based on the matchResult setting. For example, if a match is found, then the value inside the first AllowableValue element (with matchResult = "matched")is used.

SelectCase element, Layout XML Data record

Use SelectCase where the contents of an element is to be determined by one of many conditions being met.

You can also use SelectCase elements within validation messages, to enable variable values in message content. See Validation Attributes, Layout XML Data record.

Structure

Rules

Inside a SelectCase element, a number of Case elements can be specified, with each containing a test condition (this has the same syntax as the one used in the Condition element). One Else element (without a test condition) may optionally be specified also.

The test will be evaluated from top to bottom.

  • If a test condition is met, the text value of that Case element will be used as the content of that Field or FieldItem element, and the test is finished.
  • If none of the test conditions are met, the text value of the Else element (if specified) will be used instead. If no Else element is present, an empty value will be set.

Attributes

test

This attribute can contain:

  • Data IDs
    • Used to refer to the data source from the Data Template record
    • Must start with a forward slash (/) and be specified within two # characters - for example:

      "#/confirmation/action/deal/dealNumber#"

      "#/eft/cashflows/cashflow/account/bank/bicCode"

  • Message Layout IDs
    • Used to refer to an attribute of the Message element from the Layout XML Data record
    • Must start with /Message/@ followed by the attribute name, and be specified within two # characters - for example:

      "#/Message/@id#"

  • Element IDs
    • These can be used to refer to a data source from the current file.
    • The id name has to be specified within two # characters.
    • A colon (:) is used as a separator between the element and its parent one layer above (optional).

      For example, if Section has id="C" and Field has id="14F" then "#C:14F#" would obtain the value from that Field. If the colon is omitted, the program will use the first element it finds with the specified id value.

  • Comparison operators: =, >, >=, <, <=, !=
  • String manipulation functions: contains, starts-with, ends-with, string-length
  • Boolean operations: and, or, not
  • Constant strings, placed within single quotation marks - for example "'some value to be compared with'".

NOTE: All the test conditions applied to the current element must be specified as one single string.

xpath

See Data Selection (xpath), Layout XML Data record.

Repeat element, Layout XML Data record

Repeat is similar to processMethod="repeat" (see Processing Methods), except that it is used when multiple Field, Section, or SubSection elements need to be repeated as a group.

Structure

Rules

Repeat wraps one or multiple Field elements if the xpath attributes of those elements return more than one value.

CAUTION: It is assumed that all Field elements return the same number of values. If not, the values returned may be misplaced in each of the repeated blocks.

Within Repeat, Field elements can be constructed in any way, including complex elements such as Combination or PriorityGroup.

NOTE: If a Field or FieldItem element returns multiple results, and it is not under a Repeat element, only the first returned value will be used as the value of that Field or FieldItem.

Extraction element, Layout XML Data record

Use this element to take part of the data source and use it in the construction of another Field or FieldItem element.

Structure

<Extraction />

Rules

Usually used within a Field or FieldItem element, but can be used by itself.

Must include one "start..." attribute and one "end..." attribute.

Attributes

  • start_incl: Include source data from the beginning of the starting match item.
  • start_excl: Include source data from the end of starting match item.
  • end_incl: Include source data up to the end of the ending match item.
  • end_excl: Include source data up to the beginning of the ending match item.
  • xpath: Indicates where the source data will come from. Optional.

For more details, see Data Selection (xpath), Layout XML Data record.

  • start_regexp: determines whether to treat the value specified in start_incl as a regular expression* for locating the starting position during text extraction
    • Required: No
    • Values: yes, no
    • Default: no
  • end_regexp: determines whether to treat the value specified in end_incl as a regular expression* for locating the end position during text extraction
    • Required: No
    • Values: yes, no
    • Default: no

*TIP: The syntax of regular expressions here corresponds to that of Microsoft's Visual Basic for Applications (VBA).

In this example, both start_incl and end_excl are regular expressions, and the process is as follows:

  1. The start_incl regular expression (/BEN/)|(/REC/) means "find a string starting with /BEN/or/REC/".
  2. The end_incl regular expression /\w+/ means "after finding a string starting as above, continue until one or more alphanumeric characters between two slashes is found. (See Tip below)
  3. Extract and output the string between the start string found (including those characters) and the end string found (excluding those characters).
  4. Repeat Steps 1 through 3 until the end of the source data is reached.

The two strings satisfying the Extraction conditions are therefore output (the first one's end string is /ELEC/; the second one's is /ABC/, and these end strings are excluded).

TIP: Alphanumeric characters are A-Z, a-z, 0-9, and _ (underscore). \w means any one of these, \w+ means one or more of them.
