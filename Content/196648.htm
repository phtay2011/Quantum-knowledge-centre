Enterprise Workflow Framework (EWF) enhancements

This section covers enhancements to custom pages* and custom reports* you design and create in the Enterprise Workflow (EWF).

*NOTE: Now called workflow pages and workflow reports (from Release 2019.4).

View changes to a dashboard

You can now review / audit any changes made to a dashboard (a Dashboard page) configured within the Enterprise Workflow (EWF).

Open the Audit query to find information that has been added to, edited within, or deleted from an existing dashboard. The system logs the user who made the change

See more: Audit

Actions on Custom Page using Workflow grid

When creating a workflow page in the Enterprise Workflow (EWF), you can use the new Workflow grid to add and edit actions to update, delete, or create deals (as for workflow reports), or to Set Context (see below). For each action, you can specify:

  • Action Timing - when to perform the action: After Load, Before Create, After Update, and so on.

    TIP: For a sequence of actions, if one fails, the rest of the actions do not complete. If multiple actions are defined for the same timing, the actions will be performed in the specified order.

  • Prerequisite - a prerequisite condition that must hold true for the action to be performed. If false, the action will be skipped.
  • Inputs - deal data items including a Reset setting, each with an Expression used to set a value. Note that:
    • a Delete action has two mandatory items: Deal Number and Reason, and
    • a Reset value of Yes functions the same way as resetting a deal in non-custom pages.
  • Outputs (Create actions only) - the DealNumber data item and a specified Column.

    See more: Workflow page - Workflow grid

New Custom page Action: Set Context

When creating a customized page in the Enterprise Workflow (EWF), you can use the Workflow grid to add an action to set a property in the current context (the deal in the Core Module or Variant you are editing), at the specified Action Timing point.

Typical examples are when you want to set a value for an Analysis Code.

The page action is called Set Context, and:

  • it is currently supported only for Dealing modules,
  • it is available under the Update folder,
  • Settings on the action are all available properties in the context,
  • there are no outputs, and
  • when the action is performed, the passed-in settings are applied to the context.

New report action: Delete/Update a deal

You can now use the Delete and Update. actions in the Enterprise Workflow (EWF). Delete requires two mandatory parameters: the deal number and a reason. Update requires the deal number. Use these actions to delete deals in bulk or update deals in bulk through a report, or to delete or update individual deals through a customized page.

After you delete a deal, you'll see Deleted and Notes badges on the deal page. Clicking the Notes badge displays the reason.

You cannot currently delete projected cash flows.

If you delete a deal in a workflow report, you will now delete any deal that is linked to the original deal.

See more: Creating a new report page

New report action: Create GL Journal

You can now create a General Ledger Journal using an Action. There is a new item in the Create folder, Create GL Journal. Applies to both workflow pages and workflow reports.

Select actions from Sources folder via Expression Builder

When creating a workflow page in the Enterprise Workflow (EWF), you can use the Expression Builder dialog box to add an external data source from a Sources folder to an expression.

NOTE: The source referenced in the Workflow action's expression can be deleted and there is no warning message.

Using a token in deal validation

You can now use a token in a Customization page (earlier versions) or workflow report in the Enterprise Workflow Framework (EWF). One use of tokens is to add extra validation to the process of setting up a deal.

Example:

  1. In the Customization page*, create your specific page for the Dealing section, and in the Validations area, create a new validation that includes:

    Condition: [Bank] = 'Southern'

    Message: "You cannot select {[Bank]} bank."

  2. Finish setting up the EWF page.
  3. When you use this page to create a deal with "Southern" as the bank, a message appears:

    You cannot select Southern bank.

TIP: A similar procedure applies when creating a workflow report.

See more: Creating Dealing Pages from a Workflow Page

New warning message on opening a generated report

If you attempt to open a generated workflow report page, but do not have the required permissions to access to all components of the report (Data Sources, Actions, and so on), you will receive the following message:

You cannot open the page because you don't have permission to retrieve all the data and/or perform all the actions defined on the page.

To find out more, open Reporting > Message Log.

Use includedValues in the XML for AnalysisCodes

In the XML section of the Customization page, you can now use the includedValues attribute to filter an existing list of values for a named Analysis Code.

For example:

<AnalysisCodes>
<AnalysisCode name="DEALS" includedValues="e*"/>
<AnalysisCodes/>

In the Analysis Code field labeled DEALS, users will only be able to select options starting with "e" or "E".

You cannot add Analysis Codes using this method, and the values called by the includedValues attribute will always be existing values. The system ignores extra values or mis-typed values.

Custom Commodity pages: include core data

In the Customization page, when you are creating a variant Commodity Swap or Exposure dealing page, you can now:

  • Include core data - include, in the variant workflow page, deals that are created in the core page; and/or
  • Publish data to core - include, in the core page, deals that are created by the variant workflow page.

This brings Commodity Swap (CMS) and Commodity Exposure deals into line with other core modules.

See more: Commodities in Quantum Six

Detect if a Counterparty group is an ancestor

You can now find out if a Counterparty is a descendant of a Counterparty group.

To do this, the new Boolean function IsUnderCounterpartyGroup is available in the Customization page. Specify the counterparty and counterparty group to find out whether the counterparty is located under the counterparty group in the hierarchy of counterparties.

If you enter an invalid name or group, the function will return "false".

See more: Functions (Expressions in Calculated Columns).

New warning message: Report query filters

When you open a query in a workflow report page created in the Enterprise Workflow (EWF), the filter values are now validated. When the value doesn't exist due to Resource Security, changes in lookup definition, Static Data renaming or removal, and so on, the value is removed from the filter. The system will show the following warning message:

This query references filter values that no longer exist or you do not have access to. These have been removed from the query.

Editing an existing report: effects on saved queries, views, and runs

When you edit an existing workflow report in the Enterprise Workflow (EWF), the system tries to keep as much as possible of any user-defined queries and views intact.

If you:

  • add a filter to the report definition - saved queries are kept;
  • remove a filter from the report definition - the filter is removed from any saved queries and any previous runs are deleted;
  • add a column - saved views are kept, but any previous runs are deleted;
  • remove a column - saved views are removed any previous runs are deleted; or
  • change a column name - saved views are kept.

When you save your changes to the workflow report, a validation message displays only when something will be removed (query, view, or run).

Previously, editing would have created invalid data and deleted the queries, views and previous runs.

See more: Report

View changes to customized pages and reports

You can now review all changes to customized page or workflow report definitions that have been made using the Enterprise Workflow (EWF).

Open the Audit query to view information that has been added to, edited within, or deleted from the existing page or workflow report, and who made the change.

NOTES:

  • The column headings in the example above are not the default headings.
  • Changes to XML content are not logged.

    See more: Audit

Join sources using new options

When creating a workflow report in the Enterprise Workflow (EWF), you can now join sources using any of the following options:

  • Left first (all from left, first match from source) - all data from the Left Source plus matching data from the main Source,
  • Left outter (all from left, matching from source) - all data from the Left Source plus all data from the main Source,
  • Inner (only matching rows) - data from matching rows only, or
  • Left anti (only rows from the left, not in both) - data from the Left Source only; all rows in the Right Source are removed.

Once the Quantum upgrade script is run, all Left Outter joins are automatically changed to Left First.

TIP: When an Inner & Left outter joins are both used, the behavior matches that in SQL - namely, that the joins are applied in the order that they are defined. So a second join would only be applied to the result of the first join.

Aggregate within a Data Source

You can now aggregate within a Data Source, therefore allowing Actions against aggregated data.

The New Source dialog box now has All Data and Aggregated radio buttons. When Aggregated is selected, the Columns tab shows both:

  • a new Group By grid - enabling you to select one or more fields to group by (or none), and
  • the Columns grid â€“ enabling you to select fields to aggregate by.

The available Aggregate By functions depend on the Data Type (as in pivots):

Data Types

Aggregate By

Amount, Number

Count, Distinct Count, Max, Min, None, Running Sum, Sum

Reference, Date, Text

Count, Distinct Count, Max, Min, None


The Data Source returns the data aggregated by the aggregation functions and grouped by the fields. The aggregated results can be used in joins with other Data Sources - that is, only the Group By fields and selected columns can be used to define a join with other Data Sources.

Previously, you could aggregate your data by creating a Pivot view, but you would lose the ability to perform Actions against the results.

Select FX rates from a Data Source

In a workflow report created in the Enterprise Workflow Framework (EWF), you can now retrieve FX rates from an FX Rates Data Source. This means that you can display a pivot of FX rates that shows rates for several currencies by rate interval.

Source for FX rate Forward Points

You can now retrieve the FX rates' forward points from a data source by creating a workflow report in the Enterprise Workflow Framework (EWF). This means you can use a report to display a pivot of FX rates that shows rates for several currencies by rate interval.

The values returned by the FX Rate Forward Points data source are solely based on the join settings given as input. You cannot select this data source by itself as it always needs to join to another data source. The Values folder that holds this data source will therefore only show up when the report uses at least one other data source.

Filter a database view data source by its columns

When creating either a customized page or a workflow report, you can now filter a view data source by its columns, to return a subset of the results from that database view data source when the customized page or generated report is run.

Each column in the database view data source is available as a setting, and each setting has the same data type as the column. A filter can also be used as a setting.
