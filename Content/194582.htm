Bank Importer

External Cash Flows - expanded Comments

(First introduced in Release SP 6.7.2, December 2018.)

The Comments field in External Cash Flows imported into Quantum may now contain up to 999 characters. This is supported for all three External Cash Flow file formats, in Prior Day and Intraday Match query results, and in CashXplorer worksheets.

(Previously, the Comments field was limited to 40 characters.)

Bank Importer performance gains

(First introduced in Release SP 6.7.2, December 2018.)

Cash Management file import times have been significantly improved:

  • with Tracing ON - by over 80 percent (7 to 25 times faster)*, or...
  • with Tracing OFF - by 25 to 68 percent.*

These improvements result from the following changes.

  • Logging enhancements

    These include two new configuration parameters (in the BkImport.ini file) to limit Log file size:

    • RollOnLogfile specifies whether or not Log files will be closed when they reach the specified or default maximum size (if so, and a new Log file is then opened). The default setting is No.
    • MaximumLogFileSize specifies the maximum Log file size and the units (kb/mb/gb) that applies when RollOnLogfile=Yes. The default is 100mb.
  • Importing transactions from one file in parallel

    NOTE:While the Log file records statements in order, the transactions within a statement may not be in their original order.

  • Importing multiple bank files in parallel

* Times recorded during testing (with AutoMatch off).

(New from Release 6.8.0.)

Cash Management file import times have been further improved by separating out the Automatch process from the import process. This applies for both importing of files and via ECHOS - consequently, the EchosAutoMatch and PerformAutoMatchAfter configuration parameters for Bank file importing are now redundant.

(Previously, Bank Importer would not finish processing a file until the Automatch was finished.)

Validation enhancements

(First introduced in Release SP 6.7.1, August 2018.)

For the BAI parser:

  • The validation logic for the ValidateUniqueStatement parameter now includes balance date checking.
  • If validation is turned on, BAI statements will be considered a duplicate if there is already a statement with the same reference, same account number, AND same balance date. A statement will not be rejected if it has the same reference and same account number, but a different balance date.

For the MT900/910 parser:

  • Validation is performed regardless of the ValidateUniqueStatement parameter setting (this has not changed).
  • The validation logic now includes bank account.
  • An MT910/MT900 statement will be considered a duplicate if there is already a statement with the same reference and same bank account.
  • A statement will not be rejected if it has the same reference but different bank accounts.

History

Prior to this enhancement:

  • A BAI statement would have been rejected if there was already a statement with the same reference and account number, whatever the balance date.
  • A MT910/MT900 would have been rejected if there was already a statement with the same reference but a different bank account.
