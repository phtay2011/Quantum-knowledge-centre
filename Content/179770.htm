About Data Template XML Data records (EFTs)

A Data Template XMl Data record for a specific EFT message type:

  • is specified in the name attribute of the mandatory DataTemplate element in the XTC Configuration XML Data record for the applicable External Method,
  • defines Quantum data that can be retrieved to build the message content,
  • cannot be changed by users, and
  • contains XML code.

Seed Data Template records

Two seed Data Template XML Data records are provided:

  • SeedXMLDataTemp_EFTandFlows (the default for SWIFT) - specifies all available cash flow, stock flow, and related confirmation* data, and
  • SeedXMLDataTemp_EFTandFlows_InclSource- as above plus details of all source database fields.

*CAUTION: Including Confirmation data in EFT messages impacts message output performance. This is more noticeable when a large number of EFTs are processed, and depending on how much additional confirmation data is requested in the data file. Use only when necessary and the data is not available from the EFT.

If no Confirmation data is required, remove the <confirmations> section completely from the Data Template record.

Elements and Attributes

These are common to both Confirmations and EFTs.

See Data Template XML Data record elements and attributes.

General Structure

For SWIFT, the Data Template record's XML code has the following general structure.

<?xml version="1.0" encoding="UTF-8"?>
<eft>
  <messageType/>
    <currentMessage>
      <messageId/>
      <externalReference/>
      <isBatched/>
    </currentMessage>
    <reason/>
    <formattedMessage/>
    <previousMessage/>
    <eftFields>
      <parameterDetails>
        ...
      <cashflowsTable>
        ...
      <systemDate>
      <swiftTags>
        ...
    </eftFields>
    <cashflows>
        ...
    </cashflows>
    <flows>
        ...
    </flows>
    <cancellationMessageIds/>
    <deliveryProfiles>
        ...
    </deliveryProfiles>
    <eftMessages>
        ...
    </eftMessages>
    <confirmations>
        ...
    </confirmations>
</eft>

Where:

  • messageType is the SWIFT Message Type;
  • currentMessage contains:
    • messageId - the unique message ID, and
    • externalReference - the unique external reference, typically used in Tag20;
  • reason is the parameter selection for Payment/Receipt or Cancellation Messages only, and can be PAYMENTS, RECEIPTS, BOTH, or CANCELLATION;
  • formattedMessage is the GENEFT/SWIFTEFT output prior to any XML manipulation;
  • previousMessage is the previous message content, if any - for example, as used for cancellation;
  • eftFields contains:
    • parameterDetails - all parameter details: for example value dates, run number, transaction types, and so on,
    • cashflowsTable - all the details from the GENSCRIPT cashflows table - for example adhocInsKeyField, badelinsBankToBankKeyField, and so on,
    • systemDate - the System Date, and
    • swiftTags - all the currently supported Generic EFT scripting SWIFT Tags such as Tag22, Tag53a, and so on;
  • cashflows contains all details related to a particular cash flow - for example, the cash flow number, deal number, cash flow date, amount, account details, transaction type, payment category, counterparty details, ordering customer details, delivery details (delivery instructions), and details of any netted cashflows (within NetDetails elements within one netting element);
  • flows contains all details related to a particular stock flow - for example, the flow number, deal number, linked cash flow number, flow date, amount, units, account details, transaction type, payment category, security identification details, and counterparty details;
  • confirmations* (if required) contains related Confirmation data fields. For details, see the section About the Data Template Files under Confirmations.

    *CAUTION: Including Confirmation data in EFT messages impacts message output performance. This is more noticeable when a large number of EFTs are processed, and depending on how much additional confirmation data is requested in the data file. Use only when necessary and the data is not available from the EFT.

    If no Confirmation data is required, remove the <confirmations> section completely from the Data Template record.

Data Template record content

The content of the Data Template record's XML code specifies the raw data in the EFT queue, as output by an EFT Processing run.

Depending on the EFT Processing query configuration, the output may include multiple cash flows (and stock flows where applicable).

One file is output per EFT event. For example, with SWIFT:

  • one data file with information about a SWIFT MT103 message based on a single or netted cash flow;
  • one data file for batched cash flows (depending on EFT Processing run settings);
  • one data file for an MT540 based on a single stock flow; or
  • one data file for an MT543 based on a stock flow with one or more cash flows linked to it.

Types of EFT output

The Data Template XML Data record's code lists the available source data that can be used to build the content specified in the Layout record. More than a thousand various data fields can potentially be used for an EFT.

Different types of EFT data files that may be produced (depending on EFT Processing run settings) are:

  • single standalone cash flow - contains a single stand-alone cash flow,
  • multiple cash flows - contains multiple cash flows that are either netted or need to be batched,
  • same-deal cash flows - contains multiple cash flows on the same day and for the same deal number,
  • single standalone stock flow - contains a single standalone stock flow,
  • single stock flow with linked cash flows - contains a single stock flow with one or more associated cash flows, and
  • cancelation - contains a single cancelation message.
